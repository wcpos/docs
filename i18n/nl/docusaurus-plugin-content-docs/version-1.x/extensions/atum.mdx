NL: # WCPOS ATUM Integration

NL: Integrates [ATUM Multi-Inventory](https://www.stockmanagementlabs.com/addons/atum-multi-inventory/) with [WCPOS Pro](/getting-started/pro-license), enabling location-based inventory, pricing, and SKUs at the Point of Sale.

NL: ATUM Multi-Inventory lets you split a product's stock across multiple inventory locations — warehouses, retail stores, and so on. This plugin connects those ATUM locations to WCPOS Pro [stores](/stores) so each POS terminal sees the correct stock levels, prices, and SKUs for its physical location.

  </FeatureCard>
  <FeatureCard icon="store" title="NL: Flexible Pricing">
NL:     Choose pricing from WooCommerce defaults, WCPOS Pro per-store prices, or ATUM location-specific prices.
  </FeatureCard>
  <FeatureCard icon="barcode" title="NL: Location SKUs">
NL:     Optionally swap the product's main SKU for an ATUM location-specific SKU at the POS.
  </FeatureCard>
  <FeatureCard icon="sync" title="NL: Audit-Safe Stock Movement">
NL:     Orders deduct and restore stock at the correct ATUM location, with full audit trail in `atum_inventory_orders`.
  </FeatureCard>
  <FeatureCard icon="arrow-right" title="NL: Product Edit Write-Back">
NL:     POS edits to stock, price, and SKU sync back to the mapped ATUM inventory row for that location.
  </FeatureCard>
</FeatureGrid>

## Vertaald: Installation {#installation}

<Steps>
  <StepCard number="1" title="NL: Install ATUM and Multi-Inventory">

NL: Install [ATUM Inventory Management](https://wordpress.org/plugins/atum-stock-manager-for-woocommerce/) and the [ATUM Multi-Inventory add-on](https://www.stockmanagementlabs.com/addons/atum-multi-inventory/). Configure your inventory locations in ATUM.

  </StepCard>
  <StepCard number="2" title="NL: Install WCPOS ATUM Integration">

NL: Install from `WP Admin > POS > Settings > Extensions`, or download the latest release from the [GitHub releases page](https://github.com/wcpos/wcpos-atum/releases) and upload via `Plugins > Add New > Upload Plugin`.

  </StepCard>
  <StepCard number="3" title="NL: Map stores to ATUM locations" isLast>

NL: Go to `POS > Stores`, edit a store, and configure the **ATUM Inventory** sidebar section. Pick the inventory location the store should use, choose a pricing source, and optionally enable SKU overrides.

  </StepCard>
</Steps>

## Vertaald: Store Configuration {#store-configuration}

NL: The plugin adds an **ATUM Inventory** section to the WCPOS Pro store editor sidebar with three settings per store:

- NL: **Inventory Location** — which ATUM location this store pulls stock from.
- NL: **Pricing Source** — where product prices come from:
  - NL: *Default* — standard WooCommerce prices
  - NL: *WCPOS Pro* — per-store pricing set in WCPOS Pro
  - NL: *ATUM* — location-specific prices from the ATUM inventory
- NL: **SKU Override** — optionally use location-specific SKUs from ATUM instead of the product's main SKU.

## Vertaald: POS Behavior {#pos-behavior}

NL: When a store has an ATUM location assigned, the product data served to the POS is automatically adjusted:

- NL: **Stock quantities** reflect the specific location's inventory, not the aggregate WooCommerce stock.
- NL: **Stock status** is recalculated based on the location's quantity.
- NL: **Prices** come from the configured pricing source.
- NL: **SKUs** are swapped to the ATUM location SKU if override is enabled.

NL: All adjustments happen transparently through the WCPOS REST API — no changes are needed on the POS app side. Product edits made from the POS are also written back to the mapped ATUM inventory row; see [Product Edit Write-Back](#product-edit-write-back) below.

## Vertaald: Stock Management {#stock-management}

NL: For POS orders placed at stores with a mapped ATUM location, the plugin lets ATUM's native stock deduction flow handle the write — but steers it to the correct location:

1. NL: **REST payload injection.** When the POS creates or updates an order, the plugin injects a `mi_inventories` entry onto each line item so ATUM knows which location to draw from. Without this, ATUM would fall back to the main inventory.
2. NL: **Location-scoped inventory filter.** The plugin filters ATUM's candidate inventory list to only those linked to the store's mapped location term, ensuring the right one is picked on both reduction and restoration.

NL: ATUM itself performs the actual stock change on order and refund, writing rows to `atum_inventory_orders` with the real `order_id` — preserving ATUM's audit trail.

## Vertaald: Product Edit Write-Back {#product-edit-write-back}

NL: When a cashier or manager edits a product or variation from the POS, the changes sync back to the mapped ATUM inventory row for that store's location — not just the main WooCommerce product. This keeps each location's stock, price, and SKU in sync with ATUM without manual updates in `WP Admin`.

NL: The write-back is triggered on WCPOS product and variation REST updates (`POST`, `PUT`, `PATCH` to `/wcpos/v1/products/...`) that include a `store_id`. The plugin looks up the store's mapped ATUM location and updates only the inventory row for that location — other locations are untouched.

### Vertaald: What Syncs {#what-syncs}

NL: The write-back respects each store's configuration so ATUM data only changes when the store actually owns that data:

| NL: Field | NL: When it syncs | NL:
| --- | --- |
| NL: **Stock quantity** | NL: Always — every store with a mapped ATUM location keeps its location row in sync. | NL:
| NL: **Regular price / Sale price / Price** | NL: Only when the store's **Pricing Source** is set to *ATUM*. | NL:
| NL: **SKU** | NL: Only when **SKU Override** is enabled for the store. | NL:

NL: If the store uses *Default* or *WCPOS Pro* pricing, ATUM price fields are left alone so ATUM continues to serve as a reference price rather than the source of truth. The same applies to SKUs when override is off.

### Vertaald: What Doesn't Trigger Write-Back {#what-doesnt-trigger-write-back}

- NL: Product creation (only updates write back — new products fall through to WooCommerce's normal save path).
- NL: Requests without a `store_id` — the POS has to tell the plugin which location to write to.
- NL: Stores without a mapped ATUM location.
- NL: Products without an existing ATUM inventory row for the store's location — the plugin will not create new inventory rows, only update existing ones.
