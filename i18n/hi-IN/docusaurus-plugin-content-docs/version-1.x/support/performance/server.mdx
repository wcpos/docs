सर्वर प्रदर्शन सीधे आपकी WCPOS एप्लिकेशन की गति और प्रतिक्रिया समय को प्रभावित करता है। यह गाइड आपकी सर्वर प्रदर्शन को मॉनिटर, निदान और ऑप्टिमाइज़ करने में मदद करती है, जिसमें अंतर्निहित मैट्रिक्स और समस्या निवारण तकनीकें शामिल हैं।

## सर्वर मैट्रिक्स को समझना

WCPOS प्रत्येक डेटा फ़ेच ऑपरेशन (उत्पाद, आदेश, ग्राहक आदि) के साथ सर्वर प्रदर्शन मैट्रिक्स को स्वचालित रूप से एकत्र करता है। आप इन मैट्रिक्स को [लॉग](/support/logs) स्क्रीन में देख सकते हैं।

### सामान्य सर्वर मैट्रिक्स

```json
{
  "total": "24692",
  "execution_time": "76.64 ms",
  "server_load": "[15.20605469,16.16357422,16.76806641]"
}
```

**विवरण:**
- **`total`** - संसाधित रिकॉर्डों की संख्या (24,692 उत्पाद आईडी)
- **`execution_time`** - ऑपरेशन को पूरा करने में लगा समय (76.64 मिलीसेकंड)
- **`server_load`** - 1, 5, और 15 मिनट के लिए सर्वर लोड औसत

## सर्वर लोड की व्याख्या

सर्वर लोड विभिन्न समय अवधियों में औसत सिस्टम लोड का प्रतिनिधित्व करता है:

- **पहला मान** - 1-मिनट का औसत (15.21)
- **दूसरा मान** - 5-मिनट का औसत (16.16)
- **तीसरा मान** - 15-मिनट का औसत (16.77)

### लोड व्याख्या

सर्वर लोड के मान भ्रामक हो सकते हैं और उन्हें सावधानी से व्याख्यायित करना चाहिए:

:::warning लोड मान भ्रामक हो सकते हैं
सर्वर लोड औसत हमेशा प्रदर्शन के साथ सीधे सहसंबंधित नहीं होते हैं। उच्च लोड मान (15+) वाले सर्वर भी बहुत प्रतिक्रियाशील हो सकते हैं यदि इसे पर्याप्त संसाधन मिले और इसे अच्छी तरह से ऑप्टिमाइज़ किया गया हो। **लोड मानों के बजाय निष्पादन समय पर ध्यान केंद्रित करें।**
:::

**सामान्य दिशानिर्देश:**
- **CPU कोर के सापेक्ष लोड** - 8-कोर सर्वर पर 8.0 का लोड पूरी क्षमता का उपयोग दर्शाता है
- **स्थायी बनाम अस्थायी** - क्षणिक स्पाइक्स सामान्य होते हैं, स्थायी उच्च लोड समस्याओं का संकेत दे सकता है
- **प्रदर्शन अधिक महत्वपूर्ण है** - उच्च लोड वाला प्रतिक्रियाशील सर्वर धीमे लोड वाले सर्वर से बेहतर है

**ध्यान देने योग्य बातें:**
- **समय के साथ बढ़ते निष्पादन समय**
- **बिना किसी स्पष्टीकरण के लगातार बढ़ता लोड**
- **उच्च लोड और धीमे निष्पादन समय** दोनों एक साथ

## प्रदर्शन मानक

### निष्पादन समय दिशानिर्देश

| ऑपरेशन | अच्छा | स्वीकार्य | खराब | महत्वपूर्ण |
|---------|-------|-----------|------|------------|
| **उत्पाद फ़ेच** | < 100ms | 100-500ms | 500ms-2s | > 2s |
| **ऑर्डर निर्माण** | < 200ms | 200-800ms | 800ms-3s | > 3s |

### रिकॉर्ड संख्या पर विचार

निष्पादन समय को रिकॉर्ड संख्या के साथ उचित रूप से बढ़ना चाहिए:

```json
// Good scaling examples
{"total": "100", "execution_time": "15.2 ms"}    // 0.15ms per record
{"total": "1000", "execution_time": "89.4 ms"}   // 0.09ms per record
{"total": "10000", "execution_time": "234.1 ms"} // 0.02ms per record

// Poor scaling examples
{"total": "100", "execution_time": "500.0 ms"}   // 5.0ms per record
{"total": "1000", "execution_time": "8000.0 ms"} // 8.0ms per record
```

## प्रदर्शन समस्याओं का निदान

### चरण 1: लॉग की निगरानी करें

1. नेविगेशन ड्रावर से **लॉग** खोलें
2. धीमी प्रक्रिया प्रदर्शन करें (उत्पाद समन्वयन, आदेश बनाना, आदि)
3. संबंधित लॉग प्रविष्टि को देखें
4. मैट्रिक्स देखने के लिए संदर्भ विस्तारित करें

### चरण 2: मैट्रिक्स का विश्लेषण करें

**उच्च निष्पादन समय + उच्च सर्वर लोड = सर्वर संसाधन की समस्या**
```json
{
  "total": "5000",
  "execution_time": "3500.0 ms",
  "server_load": "[12.45, 11.23, 10.87]"
}
```
*समाधान: सर्वर संसाधनों में वृद्धि करें या सर्वर कॉन्फ़िगरेशन को ऑप्टिमाइज़ करें*

**उच्च निष्पादन समय + सामान्य सर्वर लोड = प्लगइन/डेटाबेस समस्या**
```json
{
  "total": "1000",
  "execution_time": "2800.0 ms",
  "server_load": "[1.23, 1.45, 1.67]"
}
```
*समाधान: धीमे प्लगइन्स की पहचान करें या डेटाबेस क्वेरीज़ को ऑप्टिमाइज़ करें*

**सामान्य निष्पादन समय + उच्च सर्वर लोड = सामान्य सर्वर ओवरलोड**
```json
{
  "total": "2000",
  "execution_time": "150.0 ms",
  "server_load": "[8.90, 9.12, 8.45]"
}
```
*समाधान: अन्य प्रक्रियाओं से सर्वर लोड को कम करें या संसाधनों को अपग्रेड करें*

## सामान्य प्रदर्शन समस्याएँ

### 1. अपर्याप्त सर्वर संसाधन

**लक्षण:**
- लगातार उच्च सर्वर लोड (> 4.0 अधिकांश सर्वरों पर)
- सभी संचालन में लंबे निष्पादन समय
- बार-बार टाइमआउट

**समाधान:**
- **CPU अपग्रेड करें** - अधिक कोर समवर्ती अनुरोधों को बेहतर तरीके से संभालते हैं
- **RAM बढ़ाएँ** - डिस्क I/O को कम करें और कैशिंग में सुधार करें
- **SSD भंडारण का उपयोग करें** - डेटाबेस प्रदर्शन में महत्वपूर्ण सुधार
- **PHP सेटिंग्स को ऑप्टिमाइज़ करें** - memory_limit, max_execution_time बढ़ाएँ

### 2. धीमी डेटाबेस क्वेरीज़

**लक्षण:**
- सामान्य सर्वर लोड के साथ उच्च निष्पादन समय
- विशेष रूप से धीमी उत्पाद/आदेश फ़ेचिंग
- लॉग में डेटाबेस से संबंधित त्रुटि कोड

**समाधान:**
- **WooCommerce HPOS सक्षम करें** - डेटाबेस प्रदर्शन में सबसे बड़ा सुधार
- **ऑब्जेक्ट कैशिंग का उपयोग करें** - Redis या Memcached यदि आपके होस्ट से उपलब्ध हो
- **WordPress को अपडेट रखें** - कोर अपडेट अक्सर डेटाबेस ऑप्टिमाइज़ेशन शामिल करते हैं
- **पोस्ट समीक्षा को सीमित करें** - wp-config.php में `define('WP_POST_REVISIONS', 3);` जोड़ें

### 3. प्लगइन हस्तक्षेप

**लक्षण:**
- प्लगइन अपडेट के बाद अचानक प्रदर्शन में गिरावट
- विशिष्ट संचालन अन्य की तुलना में बहुत धीमे
- सामान्य सर्वर लोड के साथ उच्च निष्पादन समय

**समस्या निवारण:**
1. **स्टेजिंग पर परीक्षण करें** - WooCommerce और WCPOS को छोड़कर सभी प्लगइन्स को बंद करें
2. **बेसलाइन मापें** - न्यूनतम प्लगइन्स के साथ निष्पादन समय रिकॉर्ड करें
3. **क्रमशः चालू करें** - आरोपणकर्ताओं को पहचानने के लिए एक-एक करके प्लगइन्स जोड़ें
4. **प्लगइन हुक्स की जाँच करें** - WooCommerce क्रियाओं में हुक करने वाले प्लगइन्स की तलाश करें

**सामान्य समस्याग्रस्त प्लगइन्स:**
- उत्पाद संचालन के दौरान भारी SEO प्लगइन्स
- जटिल इन्वेंटरी प्रबंधन प्रणालियां
- वास्तविक समय एनालिटिक्स/ट्रैकिंग प्लगइन्स
- खराब कोडित कस्टम प्लगइन्स

### 4. WordPress/WooCommerce कॉन्फ़िगरेशन

**लक्षण:**
- असंगत प्रदर्शन
- लॉग में मेमोरी से संबंधित त्रुटियाँ
- धीमा व्यवस्थापक डैशबोर्ड

**ऑप्टिमाइजेशन चेकलिस्ट:**
- **PHP संस्करण** - बेहतर प्रदर्शन के लिए PHP 8.0+ का उपयोग करें
- **WooCommerce HPOS** - उच्च प्रदर्शन ऑर्डर स्टोरेज सक्षम करें (नीचे देखें)
- **WordPress कैशिंग** - यदि उपलब्ध हो तो ऑब्जेक्ट कैशिंग सक्षम करें
- **WooCommerce सेटिंग्स** - उत्पाद छवि आकारों को ऑप्टिमाइज़ करें

### WooCommerce उच्च-प्रदर्शन ऑर्डर स्टोरेज (HPOS)

:::tip सबसे बड़ा प्रदर्शन लाभ
**HPOS WooCommerce के लिए प्रदर्शन में सबसे महत्वपूर्ण सुधारों में से एक है।** यह आदेशों को WordPress पोस्ट तालिका के बजाय कस्टम डेटाबेस तालिकाओं में संग्रहीत करता है, जिससे कई आदेशों वाले स्टोर के लिए प्रदर्शन में नाटकीय रूप से सुधार होता है।
:::

**लाभ:**
- **तेज़ ऑर्डर क्वेरीज़** - ऑर्डर को ऑप्टिमाइज़्ड डेटाबेस संरचना में संग्रहीत किया गया
- **घटी हुई डेटाबेस लोड** - ऑर्डर्स को पोस्ट/पृष्ठों से अलग करता है
- **बेहतर स्केलेबिलिटी** - बड़ी संख्या में ऑर्डर्स को प्रभावी ढंग से संभालता है
- **बेहतर प्रशासनिक प्रदर्शन** - तेज ऑर्डर प्रबंधन स्क्रीन

**कैसे सक्षम करें:**
1. `WooCommerce > सेटिंग्स > एडवांस्ड > फीचर्स` पर जाएं
2. "उच्च प्रदर्शन ऑर्डर स्टोरेज" सक्षम करें
3. माइग्रेशन प्रक्रिया का पालन करें

**और सीखें:**
- [WooCommerce HPOS दस्तावेज़](https://woocommerce.com/document/high-performance-order-storage/)
- [HPOS माइग्रेशन गाइड](https://woocommerce.com/document/high-performance-order-storage/#section-3)

## सर्वर निगरानी सर्वोत्तम प्रथाएं

### 1. नियमित प्रदर्शन जांच

- **साप्ताहिक समीक्षाएँ** - प्रदर्शन प्रवृत्तियों के लिए लॉग की जांच करें
- **बेसलाइन माप** - सामान्य निष्पादन समय रिकॉर्ड करें
- **पीक समय निगरानी** - उच्च ट्रैफिक अवधि के दौरान निगरानी करें

### 2. प्रदर्शन अलर्ट सेट करें

इन चेतावनी संकेतों की निगरानी करें:
- निष्पादन समय > 1000ms लगातार
- सर्वर लोड > 5.0 लंबे समय तक
- लॉग में बार-बार टाइमआउट त्रुटियाँ

### 3. क्षमता योजना

वृद्धि प्रवृत्तियों को ट्रैक करें:
- **रिकॉर्ड संख्या की वृद्धि** - उत्पाद, आदेश, ग्राहक
- **प्रदर्शन में गिरावट** - निष्पादन समय कैसे स्केल करता है
- **संसाधन उपयोग** - CPU, मेमोरी, डिस्क उपयोग

## सर्वर ऑप्टिमाइजेशन रणनीतियाँ

### 1. WordPress/WooCommerce सर्वोत्तम प्रथाएं

**HPOS सक्षम करें:**
- WooCommerce के लिए एकमात्र सबसे प्रभावी प्रदर्शन सुधार
- विवरण के लिए ऊपर HPOS अनुभाग देखें

**PHP कॉन्फ़िगरेशन (अपने होस्ट से परामर्श करें):**
```ini
memory_limit = 512M
max_execution_time = 300
max_input_vars = 3000
```

**WordPress कॉन्फ़िगरेशन:**
```php
// In wp-config.php - limit post revisions
define('WP_POST_REVISIONS', 3);

// Enable WordPress debug logging if needed
define('WP_DEBUG_LOG', true);
```

### 2. होस्टिंग-स्तरीय ऑप्टिमाइजेशन

**ऑब्जेक्ट कैशिंग:**
- अपने होस्ट से Redis या Memcached की उपलब्धता के बारे में पूछें
- कई प्रबंधित WordPress होस्ट इसे स्वचालित रूप से प्रदान करते हैं

**PHP संस्करण:**
- महत्वपूर्ण प्रदर्शन सुधार के लिए PHP 8.0+ का उपयोग करें
- अधिकांश होस्ट आसान PHP संस्करण स्विचिंग की अनुमति देते हैं

**सर्वर संसाधन:**
- सुनिश्चित करें कि RAM पर्याप्त है (न्यूनतम 1GB, अधिमानतः 2GB+)
- SSD भंडारण पारंपरिक ड्राइव की तुलना में डेटाबेस प्रदर्शन में बहुत सुधार करता है

## मदद कब मांगें

यदि:

- **सर्वर लोड लगातार > 8.0** है भले ही आप ऑप्टिमाइजेशन प्रयास कर रहे हों
- **निष्पादन समय > 5000ms** सरल ऑपरेशंस के लिए है
- लॉग में **मेमोरी त्रुटियाँ** बार-बार दिखाई दे रही हैं
- डेटाबेस क्वेरीज़ लगातार > 2 सेकंड ले रही हैं

उन्हें प्रदान करें:
- आपके लॉग से सर्वर मैट्रिक्स
- सक्रिय प्लगइन्स की सूची
- सर्वर विनिर्देश (CPU, RAM, स्टोरेज प्रकार)
- WordPress और WooCommerce संस्करण
