このページでは、キャッシャー（またはマネージャー）向けに日次締め処理の手順を説明します。レポート画面自体のリファレンス（フィルター、列、集計パネルの内容）については[レポート](/reports)を参照してください。

:::info Pro 機能
レポート画面を利用するには [WCPOS Pro](/getting-started/pro-license) が必要です。Pro がない場合、日次締め処理は **WP Admin → WooCommerce → 分析** から行う必要があります。
:::

:::note 手動の現金計数について
WCPOS には、開始釣銭準備金や「レジ開け／レジ締め」機能は組み込まれていません。シフトセッションの打刻、アプリ内のドロワー計数、自動差異計算のいずれもありません。レポート画面には、レジで **受け取るべき** 金額が表示されます。キャッシャーがドロワーの現金を数え、手動で照合します。このページでは、その手順について説明します。
:::

## 初回の締め処理の前に {#before-your-first-close}

一度決めたら、すべてのシフトで統一してください。

- **開店時の釣銭準備金。** 一般的な小売レジでは、釣銭用に少額紙幣・硬貨で$100〜$200を用意して営業を開始します。金額を決めて毎日統一しておくと、日次の精算計算がシンプルになります。
- **釣銭準備金の記録方法。** WCPOSではこの金額を管理しません。付箋、チェックシート、またはレジに常備するスプレッドシートなどを使用してください。どの方法でも、*釣銭準備金*と*予想ドロワー合計金額*を同じ記録に残しておくと、後から精算を監査できます。
- **計数担当と確認担当。** ベストプラクティスは2名体制（キャッシャーが計数し、マネージャーが確認）です。1名体制の店舗ではキャッシャーが両方を行います。これを運用ポリシーとして記録しておいてください。

## レジの開局 {#opening-the-till}

シフト開始時に以下を行います：

1. 金庫から釣銭準備金を取り出し、ドロワーに入れます。
2. 精算シートに釣銭準備金の金額と開局時刻を記入します。
3. WCPOSに正しい店舗でサインインします（マルチ店舗サイトの場合）。どの店舗を選択すべきか不明な場合は、[マルチ店舗設定](/stores)を参照してください。

以上で完了です。WCPOSがシフト中の売上をユーザーアカウントと選択した店舗に紐づけて記録します。

## シフト中の注意事項 {#during-the-shift}

閉局作業をスムーズにするために、以下の点に注意してください：

- **クーポンと明細行の値引きを一貫して使い分けてください。** どちらも臨時の調整に使用できますが、クーポンはレポートで追跡可能な値引きとして表示され、明細行の価格変更は表示されません。それぞれのメリット・デメリットについては[値引き](/pos/cart/discounts)を参照してください。
- **注文は放棄せず、保留にしてください。**「POS - Open」の注文（**サーバーに保存**で保存されたもの）は、注文一覧やレポートに明確に表示されますが、放棄されたカートは記録が残りません。お客様が戻った際に、保留中の注文を再開できます。
- **シフト中にドロワーの現金が多くなったら、余分な現金を回収してください。**回収ごとに金額と時間を記録しておくと、閉店時の精算が合います。WCPOS には回収の記録機能がないため、精算シートに記載してください。

## 閉店手順 {#the-close-procedure}

<Steps>
  <StepCard number="1" title="売上の受付を停止する">

入口のドアを施錠するか、受付を一時停止して、計算中に新しい取引が発生しないようにします。閉店作業中にお客様が来店された場合は、計算とレポートを完了してから会計してください。レポート期間の**後に**売上を追加すると、合計が合わなくなります。

  </StepCard>
  <StepCard number="2" title="ドロワーの現金を数える">

ドロワーをバックオフィスまたは静かなカウンターに持って行き、すべての金種を数えます。合計を記録してください。これが**実計現金額**です。

シフト中に現金の回収や現金での支払い（返金、レジからの仕入先への支払いなど）を行った場合は、それらの金額も準備しておいてください。

  </StepCard>
  <StepCard number="3" title="レポート画面を開く">

POS で**レポート**を開きます。フィルターを設定して、レポートがシフトの時間帯を正確にカバーするようにします：

- **日付範囲：***今日*（後日精算する場合は該当する日付を指定）
- **レジ担当者：**複数のレジ担当者がレジを使用し、自分の売上のみを確認したい場合は、自分のユーザーを選択します
- **店舗:** 現在の店舗が自動設定されます
- **ステータス:** デフォルトのまま（通常は*完了*＋*処理中*）にしておきます

右側の売上サマリーパネルはフィルターを変更すると更新されます。*決済方法*の下にある**現金**の金額に注目してください。これが**現金売上の期待値**です。

  </StepCard>
  <StepCard number="4" title="現金の照合">

以下の計算を行います：

> **ドロワー期待残高 ＝ オープニングフロート ＋ 現金売上 − 現金返金 − 現金回収／払出**

**実際に数えた現金**と比較します。一致するはずです。

- **一致した場合:** 問題ありません。精算シートに署名・日付を記入し、現金を回収すれば現金の照合は完了です。
- **不一致の場合:** 署名する前に下記の[差異の処理](#variance-handling)を確認してください。

</StepCard>
  <StepCard number="5" title="カード決済の照合">

次にカード決済を確認します。売上サマリーから**カード**の金額（および Stripe Terminal、SumUp、Vipps/MobilePay などの統合ゲートウェイの金額）を確認してください。

カード端末で**バッチ合計**を実行します（メニューの場所は端末によって異なりますが、通常はデバイスの*レポート*または*締め処理*にあります）。以下を比較してください：

- 同一期間の**端末バッチ合計**と**WCPOSカード合計**を比較します。両者は一致するはずです。
- 手動バッチ精算の端末を使用している場合は、帰宅前に**今すぐバッチを精算してください**。Stripe Terminalなどの統合型ゲートウェイは自動精算されます。Webチェックアウト経由のSquare設定については、[Webチェックアウト](/payment/custom-gateways/web-checkout)を参照してください。

</StepCard>
  <StepCard number="6" title="レポートを印刷する">

レポート画面のサマリーパネルで**印刷**をクリックします。通常、設定済みの印刷ルーティング（[印刷ルーティング](/templates/printer-setup#print-routing)を参照）を使用してレシートプリンターまたはドキュメントプリンターにレポートが送信され、紙の記録が作成されます。

印刷されるレポートには、店舗情報、期間、項目別合計、支払方法の内訳、および（注文テーブルで注文を選択している場合は）注文ごとの詳細が含まれます。

  </StepCard>
  <StepCard number="7" title="現金を回収して締める" isLast>

- **翌日用の釣銭準備金**をドロワーに戻します。
- 残りの現金は、署名済みのレジ精算シートと印刷したレポートをホチキスで留めたものと一緒に、金庫投入用バッグに入れます。
- 使用している外部システム（スプレッドシート、会計ソフトウェアなど）を更新します。
- WCPOSアプリを閉じて施錠します。

</StepCard>
</Steps>

## 差異の対処 {#variance-handling}

現金の実数が予想額と一致しない場合は、締め作業の前に以下の手順を順番に確認してください。

1. **ドロワーを再度数えてください。** ミスは起こります。特に硬貨を中心に、もう一度数え直してください。
2. **記録漏れの現金取引がないか確認してください。** 別のデバイスやブラウザタブで売上を処理し、この期間に含まれていないものはありませんか？レポートの日付範囲とレジ担当者フィルターを再確認してください。
3. **保留中／未完了の注文がないか確認してください。** 注文を *POS - Open* ステータスでフィルタリングします。入力済みで未会計の売上がここに表示されます。今すぐ完了するか、差異が生じることを想定してください。
4. **返金と払い出し。** 本日の現金返金および現金払い出しがすべてシートに金額とともに記録されていることを確認してください。
5. **クーポンと割引。** 適用された割引が合計に反映されていることを確認してください。レポートの*合計割引額*の行には追跡対象のクーポン割引が表示されます。明細の価格変更は*純売上*に直接反映されます（[割引](/pos/cart/discounts#what-changed-in-v190)を参照）。
6. **おつりの渡し間違い。** 差異の最も一般的な原因です。少額の差異（5ドル未満）は通常これが原因です。大きな差異は調査が必要です。

原因が判明したかどうかにかかわらず、差異をシートに記録してください。小さな差異でも記録として残しておく方が、追跡されない不一致よりも望ましいです。

## 複数レジ担当者・複数店舗のシナリオ {#multi-cashier-or-multi-store-scenarios}

<Accordion>
  <AccordionItem question="今日は複数のレジ担当者が同じレジを使用した">

2つの方法があります：

- **レジ担当者ごとに締め処理**（最も正確）：各レジ担当者が締め処理の前にレポートを自分のユーザーでフィルタリングします。各自が受け取った金額を数え、自分の担当分を照合します。ドロワーはシフト間で物理的に引き継ぐことができ、各引き継ぎが簡易的な締め処理となります。
- **1日分をまとめて締め処理**：レジ担当者フィルターを*全員*のままにし、営業終了時にドロワーを1回照合します。速いですが、差異の原因を特定しにくくなります。

レポート画面のレジ担当者フィルターは内訳表示に使用されるフィルターと同一です。レジ担当者別の個別パネルはありません。

  </AccordionItem>

  <AccordionItem question="今日は1人のレジ担当者が複数の店舗で勤務した">

一方の店舗からサインアウトし、もう一方の店舗にサインインします。各セッションは `_pos_store` 注文メタを通じて正しい店舗に記録されます。閉店時には、レポート画面の**店舗**フィルターを切り替えて、店舗ごとにレジ精算を行います。

一日の終わりにサイト全体を締める場合は、店舗フィルターを*すべて*に設定して全体の締め処理も行います。これは店舗別の締めをクロスチェックするのに便利です。

  </AccordionItem>

  <AccordionItem question="店舗ごとに別の担当者が締め処理を行う">

これが通常のパターンです。各店舗の閉店担当者が自分の店舗にサインインし、その店舗と当日の日付範囲でフィルターしたレポートを実行して、その場所のレジ精算を行います。サイトオーナーは翌朝 **WP Admin → WooCommerce → 分析** からすべてのレポートを確認できます。分析画面では全店舗のデータが集約されます。

  </AccordionItem>

  <AccordionItem question="締め処理を開始した後に売上が発生した">

今回の締めには含めないでください。対応方法は次の2つです：

- **レポート期間を延長して**その売上を含める（計算済みの現金にもその売上分を加算します）。
- **翌日のレポートに回す** — 遅れた売上の現金は翌日のオープニングシートに繰越として記録します。

ルールを決めたら一貫して守りましょう。方法を途中で変えると、帳簿の監査が難しくなります。

  </AccordionItem>
</Accordion>

## ベストプラクティス {#best-practices}

- **毎日締め処理を行いましょう。** 1日でもスキップすると2日分の差異を追跡する必要があり、はるかに困難になります。
- **売上が少ない日でもレポートを印刷して保管しましょう。** 監査や紛争の際に身を守るのは紙の記録です。
- **ユーザーアカウントを共有しないでください。** 各レジ担当者が自分専用の WordPress ユーザーを使うことで、担当者フィルターによる個人別レポートが正しく機能します。
- **毎週 WooCommerce の分析と照合しましょう** (`WP Admin → WooCommerce → Analytics`) — 同じデータを異なる形式で表示します。合計が一致しない場合、通常は保留中の注文が遅れて完了したか、返金が POS ではなく WP Admin で処理されたことが原因です。
- **釣銭準備金を一定に保つこと。** 開始時の釣銭準備金を頻繁に変更すると、計算ミスの原因になります。
