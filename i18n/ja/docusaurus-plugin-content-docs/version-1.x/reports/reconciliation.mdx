このページでは、レジ担当者（または管理者）が終業時の締め処理を行う手順を説明します。レポート画面自体のリファレンス（フィルター、列、概要パネルに含まれる内容）については、[レポート](/reports)を参照してください。

:::info Pro機能
レポート画面には[WCPOS Pro](/getting-started/pro-license)が必要です。Proがない場合、終業時の照合は**WP Admin → WooCommerce → アナリティクス**から行う必要があります。
:::

:::note 手動の現金カウント
WCPOSには、釣銭準備金や「レジを開く / レジを締める」機能は組み込まれていません。出退勤用のシフトセッション、アプリ内でのドロワー集計、自動の差異計算はありません。レポート画面には、レジで**本来**受け取っているはずの金額が表示されます。レジ担当者はドロワーを数え、手動で比較します。このページでは、その周辺の手順を説明します。
:::

## 初回の締め処理の前に {#before-your-first-close}

一度決めたら、シフトをまたいで同じ運用を続けてください:

- **開始時の釣銭準備金額。** 一般的な小売店のレジでは、釣銭用に小額紙幣や硬貨で$100–$200を用意して営業を開始します。金額を決めて一貫して運用すると、日次の照合作業の計算が簡単になります。
- **釣銭準備金を記録する場所。** WCPOSはこれを追跡しません。レジに置いておく付箋、クリップボードの用紙、またはスプレッドシートが必要です。どれを選ぶ場合でも、後で照合を監査できるように、*開始時の釣銭準備金*と*想定ドロワー合計*の両方を同じ記録に残します。
- **誰が数え、誰が署名するか。** 2人で行うのがベストプラクティスです（キャッシャーが数え、マネージャーが確認）。スタッフが1人だけの店舗ではこの手順を省略し、キャッシャーが両方を行います。その旨を運用方針として記録してください。

## レジを開く {#opening-the-till}

シフト開始時に行うこと:

1. 金庫から釣銭準備金を取り出し、ドロワーに入れます。
2. 照合シートに釣銭準備金額と開始時刻を記入します。
3. 正しい店舗（複数店舗サイトの場合）でWCPOSにサインインします。どれを選ぶべきか不明な場合は、[複数店舗設定](/stores)を参照してください。

以上です。WCPOSは、シフト中の売上をユーザーアカウントと選択した店舗に紐づけて記録するようになります。

## シフト中 {#during-the-shift}

締め作業を簡単にするために、次の点に注意してください:

- **クーポンまたは明細単位の割引を一貫して使用します。** どちらもその場での調整に使用できます。クーポンはレポートで追跡対象の割引として表示されますが、明細単位の価格変更は表示されません。違いについては[割引](/pos/cart/discounts)を参照してください。
- **注文を放棄せず、保留にします。** **サーバーに保存**で保存された「POS - オープン」の注文は、注文とレポートに明確に表示されます。放棄されたカートは記録に残りません。顧客が戻ってきたら、保留した注文を後で再開できます。
- **シフト中にドロワーの現金が多くなりすぎたら、余剰分を抜きます。** 締め時に計算が合うよう、現金を抜くたびに金額と時刻を記録します。WCPOS は現金抜きを記録しないため、照合シートで管理します。

## 締め手順 {#the-close-procedure}

<Steps>
  <StepCard number="1" title="販売受付を停止する">

集計中に新しい取引が入らないよう、入口を施錠するか、待ち行列を一時停止します。締め作業中に顧客が来店した場合は、会計を行う**前に**集計とレポートを完了してください。レポート期間の後に販売を追加すると、合計が合わなくなります。

  </StepCard>
  <StepCard number="2" title="ドロワーを数える">

ドロワーをバックオフィスまたは静かなカウンターに移し、すべての金種を数えます。合計をメモします。これが**実際に数えた現金**です。

シフト中に現金を抜いたり、レジから現金を支払ったりした場合（返金、仕入先への現金支払いなど）は、その金額も手元に用意しておきます。

  </StepCard>
  <StepCard number="3" title="レポート画面を開く">

POS で**レポート**に移動します。レポートがシフトを正確に対象にするよう、フィルターを設定します。

- **日付範囲:** *今日*（後から照合する場合は対象の日付）
- **レジ担当者:** 複数のレジ担当者が同じレジを使用していて、自分の販売だけを確認したい場合は自分のユーザー
- **店舗:** 現在の店舗に自動設定されます
- **ステータス:** デフォルトのままにします（通常は *完了* + *処理中*）

フィルターを変更すると、右側の売上サマリーパネルが更新されます。*支払い方法* の **現金** の金額を確認してください。これが **予想される現金売上** です。

  </StepCard>
  <StepCard number="4" title="現金を照合する">

計算を行います:

> **予想されるドロワー合計 = 開始時の釣銭 + 現金売上 − 現金返金 − 現金ドロップ/支払い**

**実際に数えた現金** と比較します。一致している必要があります。

- **完全に一致:** 問題ありません。シートに署名と日付を記入し、現金をドロップすれば、現金の処理は完了です。
- **不一致:** 署名する前に、下の [差異の処理](#variance-handling) を確認してください。

</StepCard>
  <StepCard number="5" title="カードを照合する">

次にカード決済を確認します。売上サマリーの **カード** の金額（および統合ゲートウェイの金額があれば、Stripe Terminal、SumUp、Vipps/MobilePay など）を確認してください。

カード端末で **バッチ合計** を実行します（正確なメニューは端末によって異なりますが、通常は端末の *レポート* または *営業終了* にあります）。比較します:

- 同じ期間の**端末バッチ合計**と**WCPOS カード合計**を比較します。一致している必要があります。
- 手動でバッチを決済する端末を使用している場合は、帰宅前に**今すぐバッチを決済**してください。Stripe Terminal などの統合ゲートウェイでは自動的に決済されます。ウェブチェックアウト経由での Square 設定については、[ウェブチェックアウト](/payment/gateways/web-checkout) を参照してください。

</StepCard>
  <StepCard number="6" title="レポートを印刷する">

レポート画面の概要パネルで、**印刷**をクリックします。通常は設定済みの印刷ルーティング（[印刷ルーティング](/hardware/printers#print-routing) を参照）を使用して、レポートをレシートプリンターまたはドキュメントプリンターに送信し、紙の記録を作成します。

印刷されたレポートには、店舗情報、期間、行ごとの合計、支払い方法の内訳、および（注文テーブルで注文を選択している場合は）注文ごとの詳細が含まれます。

  </StepCard>
  <StepCard number="7" title="現金を投函して承認する" isLast>

- **明日の開始釣銭**をドロワーに戻します。
- 残りの現金は、署名済みの照合シートと印刷済みレポートをホチキス留めして、金庫投函用バッグに入れます。
- 使用している外部システム（スプレッドシート、会計ソフトウェアなど）を更新します。
- WCPOS アプリを閉じ、施錠します。

</StepCard>
</Steps>

## 差異の処理 {#variance-handling}

実際に数えた現金が想定額と一致しない場合は、承認する前に次の順序で確認してください:

1. **ドロワーを数え直します。** 間違いは起こります。特に硬貨をもう一度数えてください。
2. **記録漏れの現金取引を確認します。** 別の端末やブラウザータブで販売を処理し、この期間に含まれていないものはありませんか。レポートの日付範囲とレジ担当者フィルターを再確認してください。
3. **保留中またはオープンの注文を確認します。** 注文を *POS - オープン* ステータスで絞り込みます。会計処理されたものの未払いの販売はここに表示されます。今完了するか、その差異を見込んでください。
4. **返金と支払い。** 今日の現金返金と現金支払いがすべて、金額とともにシートに記録されていることを確認してください。
5. **クーポンと割引。** 適用された割引が合計に反映されていることを確認してください。レポートの *割引合計* 行には追跡されたクーポン割引が表示されます。明細項目の価格上書きは *純売上* を直接減らします ([割引](/pos/cart/discounts#what-changed-in-v190) を参照)。
6. **お釣りの渡し間違い。** 差異の最も一般的な原因です。少額の差異 ($5 未満) は通常これに該当します。大きな差異は調査が必要です。

原因が分かったかどうかにかかわらず、差異をシートに記録してください。小さな差異でも記録を残しておく方が、追跡されない不一致よりも望ましいです。

## 複数レジ担当者または複数店舗のシナリオ {#multi-cashier-or-multi-store-scenarios}

<Accordion>
  <AccordionItem question="今日、複数のレジ担当者が同じレジを使用した">

選択肢は 2 つあります。

- **レジ担当者ごとに 1 回締める** (最も正確): 各レジ担当者は締め処理の前に、レポートを自分のユーザーで絞り込みます。それぞれが受け取った金額を数え、自分の担当分を照合します。ドロワーはシフト間で物理的に引き継げます。各引き継ぎは小さな締め処理になります。
- **1 日全体で 1 回締める**: レジ担当者フィルターを *すべて* のままにし、営業終了時にドロワーを 1 回だけ照合します。より速く処理できますが、差異の原因を特定しにくくなります。

レポート画面のレジ担当者フィルターは、内訳の表示に使われるものと同じです。レジ担当者別のパネルはありません。

  </AccordionItem>

  <AccordionItem question="今日、1人のレジ担当者が複数の店舗で勤務した">

一方の店舗からサインアウトし、もう一方の店舗でサインインします。各セッションは `_pos_store` 注文メタによって正しい店舗に記録されます。締め処理時には、レポート画面の **店舗** フィルターを切り替えて、店舗ごとに照合を行います。

1日の終わりにサイト全体を締める場合は、店舗フィルターを *すべて* に設定して全体の締め処理も行います。店舗ごとの締め結果を照合するのに役立ちます。

  </AccordionItem>

  <AccordionItem question="店舗ごとに締め担当者が異なる">

これは通常の運用です。各店舗の締め担当者が自店舗でサインインし、その店舗と当日の日付範囲で絞り込んだレポートを実行して、その場所のキャッシュドロワーを照合します。サイト所有者は翌朝、店舗全体を集計する **WP Admin → WooCommerce → アナリティクス** からすべて確認できます。

  </AccordionItem>

  <AccordionItem question="締め処理を開始した後に販売が発生した">

今回の締め処理には含めないでください。次のいずれかを行います。

- **レポート期間を後ろに延ばして**その販売を含めます（新しい販売分を含めるように現金の実数も調整します）。または、
- **明日のレポートに回し**、遅れて発生した販売の現金を明日の開始シートに繰越として記録します。

運用方針を1つ選び、それを継続してください。方法を切り替えると、帳簿の監査が難しくなります。

  </AccordionItem>
</Accordion>

## ベストプラクティス {#best-practices}

- **毎日締め処理を行ってください。** 1日でも省略すると、2日分にまたがる差異を追跡することになり、はるかに困難になります。
- **静かな日でもレポートを印刷して保管してください** — 監査や異議申し立ての際に身を守るのは、紙の記録です。
- **ユーザーアカウントを共有しないでください。** レジ担当者ごとに個別の WordPress ユーザーを使用すると、レジ担当者フィルターが個人別レポートで正しく機能します。
- **WooCommerce Analytics と週次で照合してください** (`WP Admin → WooCommerce → Analytics`) — 同じデータを別の表示形式で確認できます。合計が一致しない場合、通常は保留注文が後から完了されたか、返金が POS ではなく WP Admin で処理されたことを意味します。
- **釣銭準備金を一定に保ってください。** 開始時の釣銭準備金を頻繁に変更すると、計算ミスを招きます。
