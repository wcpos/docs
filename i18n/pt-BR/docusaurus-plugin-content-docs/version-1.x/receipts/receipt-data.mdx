Os modelos HTML sem lógica e os modelos XML térmicos do WCPOS são renderizados a partir da mesma carga útil canônica de recibo. Use caminhos com ponto do Mustache, como `{{order.number}}`, `{{store.name}}` e `{{totals.total_display}}`. Os arrays são renderizados com seções:

```mustache
{{#lines}}
  {{name}} x {{qty}} — {{line_total_display}}
{{/lines}}
```

O contrato canônico é produzido no servidor pelo construtor de dados de recibo do WCPOS e espelhado pelo renderizador de recibos offline no aplicativo. Os recibos são abertos imediatamente a partir dos dados locais e, quando disponível, são atualizados com a resposta do servidor; portanto, os modelos personalizados devem usar os campos abaixo em vez de métodos de pedido PHP.

Os dados de recibo v1.1 adicionam campos registrados de preço normal, preço de venda antes do cupom e economia para cada linha de produto. Também fornecem totais de economia no nível do pedido, incluindo um valor de **Total economizado** que combina a economia do preço normal com os descontos do WooCommerce sem alterar os totais ou relatórios do pedido.

## Regras de renderização {#rendering-rules}

### Campos de moeda {#currency-fields}

Os campos numéricos de valores monetários são preservados como números, e o renderizador adiciona campos `_display` adaptados à localidade para a saída do modelo:

| Campo numérico | Campo de exibição |
|---------------|---------------|
| `totals.total` | `totals.total_display` |
| `lines[].line_total` | `lines[].line_total_display` |
| `payments[].amount` | `payments[].amount_display` |
| `tax_summary[].tax_amount` | `tax_summary[].tax_amount_display` |

Prefira campos `_display` ao imprimir recibos. Use campos numéricos apenas em seções condicionais ou em saídas legíveis por máquina.

### Campos compatíveis com a exibição de impostos {#tax-display-aware-fields}

Diversos campos têm variantes com e sem impostos, além de um valor de conveniência para exibição. O valor de conveniência segue a configuração da loja para exibição de impostos no carrinho.

| Campo de conveniência | Campo com impostos | Campo sem impostos |
|-------------------|-----------------|-----------------|
| `lines[].regular_price` | `lines[].regular_price_incl` | `lines[].regular_price_excl` |
| `lines[].selling_price` | `lines[].selling_price_incl` | `lines[].selling_price_excl` |
| `lines[].unit_savings` | `lines[].unit_savings_incl` | `lines[].unit_savings_excl` |
| `lines[].line_regular_total` | `lines[].line_regular_total_incl` | `lines[].line_regular_total_excl` |
| `lines[].line_selling_total` | `lines[].line_selling_total_incl` | `lines[].line_selling_total_excl` |
| `lines[].line_savings` | `lines[].line_savings_incl` | `lines[].line_savings_excl` |
| `lines[].unit_price` | `lines[].unit_price_incl` | `lines[].unit_price_excl` |
| `lines[].unit_subtotal` | `lines[].unit_subtotal_incl` | `lines[].unit_subtotal_excl` |
| `lines[].line_subtotal` | `lines[].line_subtotal_incl` | `lines[].line_subtotal_excl` |
| `lines[].discounts` | `lines[].discounts_incl` | `lines[].discounts_excl` |
| `lines[].line_total` | `lines[].line_total_incl` | `lines[].line_total_excl` |
| `fees[].total` | `fees[].total_incl` | `fees[].total_excl` |
| `shipping[].total` | `shipping[].total_incl` | `shipping[].total_excl` |
| `discounts[].total` | `discounts[].total_incl` | `discounts[].total_excl` |
| `totals.subtotal` | `totals.subtotal_incl` | `totals.subtotal_excl` |

| `totals.total` | `totals.total_incl` | `totals.total_excl` |

### Objetos de data {#date-objects}

Os campos de data são objetos com várias variantes pré-formatadas. Isso evita a formatação de datas dentro do Mustache.

| Campo | Descrição |
|-------|-------------|
| `datetime`, `date`, `time` | Strings padrão de data/hora |
| `datetime_short`, `datetime_long`, `datetime_full` | Formatos combinados que consideram a localidade |
| `date_short`, `date_long`, `date_full` | Formatos somente de data que consideram a localidade |
| `date_ymd`, `date_dmy`, `date_mdy` | Formatos de data com ordem fixa |
| `weekday_short`, `weekday_long` | Nomes dos dias da semana |
| `day`, `month`, `month_short`, `month_long`, `year` | Partes individuais da data |

Objetos de data disponíveis: `order.created`, `order.paid`, `order.completed`, `order.printed` e `refunds[].date`. `order.printed` é atualizado no momento da renderização, o que é útil para reimpressões.

### itens {#lines}

Use um loop com `{{#lines}}...{{/lines}}`.

| Campo | Tipo | Descrição |
|-------|------|-------------|
| `key` | cadeia de caracteres | Chave estável da linha/ID do item do pedido |
| `sku` | cadeia de caracteres | SKU do produto |
| `name` | cadeia de caracteres | Nome de exibição do produto ou da linha |
| `qty` | número | Quantidade vendida |
| `qty_refunded` | número | Quantidade reembolsada desta linha |
| `regular_price` / `_incl` / `_excl` | número/nulo | Preço unitário regular registrado |
| `selling_price` / `_incl` / `_excl` | número/null | Preço unitário de venda antes do cupom — o preço registrado no POS quando disponível; caso contrário, derivado do subtotal da linha armazenado |
| `unit_savings` / `_incl` / `_excl` | número/nulo | `regular_price - selling_price`, com mínimo de zero |
| `line_regular_total` / `_incl` / `_excl` | número/nulo | Preço regular registrado multiplicado pela quantidade |
| `line_selling_total` / `_incl` / `_excl` | número/nulo | Preço de venda antes dos cupons multiplicado pela quantidade |
| `line_savings` / `_incl` / `_excl` | número/nulo | Economia total entre o preço regular e o preço de venda desta linha |
| `savings_in_discounts` | booleano | Verdadeiro quando um pedido legado já inclui a mesma economia em `discounts` |
| `unit_subtotal` / `_incl` / `_excl` | número | Valor unitário do WooCommerce antes dos descontos; não use como preço regular |
| `unit_price` / `_incl` / `_excl` | número | Valor unitário final após os descontos do WooCommerce |
| `line_subtotal` / `_incl` / `_excl` | número | Subtotal da linha no WooCommerce antes dos descontos |
| `discounts` / `_incl` / `_excl` | número | Desconto nativo da linha do WooCommerce como valor positivo; normalmente apenas de cupom em pedidos atuais |
| `line_total` / `_incl` / `_excl` | número | Total final da linha |
| `total_refunded` | número | Total reembolsado para esta linha como valor positivo |
| `taxes` | matriz | Linhas de imposto por alíquota para esta linha |
| `meta` | matriz | Metadados do item do pedido como pares `{key, value}` |
| `attributes` | matriz | Atributos do produto/variação como pares `{key, value}` |

Cada campo monetário tem um correspondente `_display` formatado, incluindo `regular_price_display`, `selling_price_display`, `unit_savings_display`, `line_regular_total_display`, `line_selling_total_display` e `line_savings_display`. Os campos com impostos incluídos e excluídos também têm variantes `_display`. Quando um campo de economia anulável é `null`, seu correspondente `_display` é totalmente omitido.

Os campos de economia são `null` quando o pedido não contém dados de preço do POS registrados suficientes. Portanto, os modelos devem envolvê-los em seções Mustache, em vez de presumir que todos os pedidos históricos têm um preço regular.

### Exibição do preço normal e da economia {#displaying-regular-price-and-savings}

Para um layout de preço unitário, exiba o preço normal registrado, o preço de venda e a economia desta forma:

```mustache
{{#lines}}
  {{#unit_savings}}
    <span style="text-decoration: line-through;">{{regular_price_display}}</span>
    <span>{{selling_price_display}}</span>
    <span>{{i18n.savings}}: -{{unit_savings_display}}</span>
  {{/unit_savings}}
{{/lines}}
```

Use os campos de total da linha quando a quantidade for relevante:

```mustache
{{#lines}}
  {{#line_savings}}
    <span style="text-decoration: line-through;">{{line_regular_total_display}}</span>
    <span>{{i18n.savings}}: -{{line_savings_display}}</span>
  {{/line_savings}}
{{/lines}}
```

`selling_price` é o preço antes dos cupons. Use `unit_price` ou `line_total` para o valor final após os descontos do WooCommerce.

:::note Reimpressão de pedidos criados antes da v1.9.0
Pedidos WCPOS mais antigos podem já incluir a redução do preço normal para o preço de venda em `discounts`. Se o seu modelo personalizado imprimir tanto as economias da linha quanto os descontos da linha, use `savings_in_discounts` para evitar mostrar a mesma redução duas vezes:

```mustache
{{#lines}}
  {{#line_savings}}
    <span style="text-decoration: line-through;">{{line_regular_total_display}}</span>
    {{^savings_in_discounts}}
      <span>{{i18n.savings}}: -{{line_savings_display}}</span>
    {{/savings_in_discounts}}
  {{/line_savings}}
  {{#discounts}}
    <span>{{i18n.discount}}: -{{discounts_display}}</span>
  {{/discounts}}
{{/lines}}
```

Se o modelo não imprimir uma linha de `discounts` no nível da linha, continue mostrando `line_savings` mesmo quando `savings_in_discounts` for verdadeiro. Os modelos incluídos tratam essa distinção automaticamente.
:::

### totais {#totals}

| Campo | Tipo | Descrição |
|-------|------|-------------|
| `totals.subtotal` / `_incl` / `_excl` | número | Subtotal do pedido antes dos descontos |
| `totals.discount_total` / `_incl` / `_excl` | número | Total de descontos do pedido como valor positivo |
| `totals.savings_total` / `_incl` / `_excl` | número/nulo | Total das economias registradas entre o preço normal e o preço de venda nas linhas de produtos, incluindo vendas de catálogo e alterações de preço no caixa |
| `totals.total_saved` / `_incl` / `_excl` | número/nulo | Descontos combinados do WooCommerce e economias no preço normal, com sobreposição legada removida |
| `totals.total_saved_complete` | booleano | Indica se o recibo tem dados de preço registrados suficientes para calcular os totais completos de economia |
| `totals.tax_total` | número | Valor total de impostos |
| `totals.total` / `_incl` / `_excl` | número | Total geral do pedido |
| `totals.paid_total` | número | Valor pago/aplicado |
| `totals.change_total` | número | Troco devolvido ao cliente |
| `totals.refund_total` | número | Total reembolsado como valor positivo |
| `totals.net_total` | número | `total - refund_total`, com mínimo de zero |
| `totals.total_qty` | número | Soma das quantidades dos itens de linha |
| `totals.line_count` | número | Número de linhas de produtos |

As variantes formatadas incluem `subtotal_display`, `discount_total_display`, `savings_total_display`, `total_saved_display`, `tax_total_display`, `total_display`, `paid_total_display`, `change_total_display`, `refund_total_display` e `net_total_display`, além de variantes inclusivas/exclusivas quando aplicável.

Use `total_saved` quando quiser um único valor voltado ao cliente para tudo o que foi economizado. Não o calcule somando `discount_total` e `savings_total` no modelo: pedidos criados antes da v1.9.0 podem já incluir a economia de preço do POS no total de descontos do WooCommerce. O campo canônico `total_saved` detecta e remove essa sobreposição.

Os totais de economia são `null` e `total_saved_complete` é falso quando alguma linha de produto não tem dados de preço registrados suficientes para um agregado confiável. Proteja a linha com `total_saved_complete` e use `total_saved` para ocultar um valor zero:

```mustache
{{#totals.total_saved_complete}}
  {{#totals.total_saved}}
    <span>{{i18n.total_saved}}: {{totals.total_saved_display}}</span>
  {{/totals.total_saved}}
{{/totals.total_saved_complete}}
```

Os modelos incluídos que exibem preços usam a mesma estrutura de verificação com as variantes com impostos incluídos (`total_saved_incl` e `total_saved_incl_display`) e mostram **Total economizado** somente quando o valor completo é maior que zero.

### tax e tax_summary {#tax-and-tax_summary}

| `transaction_id` | texto | ID da transação do gateway |
| `tendered` | número | Valor em dinheiro entregue, quando presente |

  </AccordionItem>

  <AccordionItem question="refunds — registros de reembolso">

### reembolsos {#refunds}

Use um loop com `{{#refunds}}...{{/refunds}}`. Os valores de reembolso são positivos; os modelos definem se devem adicionar um sinal de menos ou renderizar um bloco separado de itens devolvidos.

| Campo | Tipo | Descrição |
|-------|------|-------------|
| `id` | número | ID do registro de reembolso |
| `date` | objeto de data | Data de criação do reembolso |
| `amount` | número | Total do reembolso |
| `subtotal` | número | Subtotal das linhas reembolsadas |
| `tax_total` | número | Imposto reembolsado |
| `shipping_total` | número | Valor do frete reembolsado |
| `shipping_tax` | número | Imposto do frete reembolsado |
| `reason` | cadeia de caracteres | Motivo do reembolso |
| `refunded_by_id` | número/nulo | ID do usuário que emitiu o reembolso |
| `refunded_by_name` | cadeia de caracteres | Nome de exibição do usuário que emitiu o reembolso |
| `refunded_payment` | booleano | Indica se o pagamento foi reembolsado pelo gateway |
| `destination` | cadeia de caracteres | `original_method`, `cash` ou `manual` |
| `gateway_id` | texto | ID do gateway usado para o reembolso |
| `gateway_title` | texto | Título de exibição do gateway |
| `processing_mode` | texto | Modo de processamento do provedor/manual |
| `lines` | matriz | Linhas de produtos reembolsados |
| `fees` | matriz | Linhas de taxas reembolsadas |
| `shipping` | matriz | Linhas de frete reembolsadas |

Os campos da linha de reembolso incluem `name`, `sku`, `qty`, `total`, `total_incl`, `total_excl`, `line_total`, `unit_total` e `taxes`. As linhas de taxas e frete do reembolso usam `label`, `total`, `total_incl`, `total_excl` e `taxes`. Variantes de exibição são adicionadas para totais e valores de impostos.

  </AccordionItem>

  <AccordionItem question="i18n — rótulos traduzidos">

### i18n {#i18n}

Use rótulos `i18n` em vez de codificar texto diretamente sempre que possível:

```mustache
{{i18n.order}} #{{order.number}}
{{i18n.cashier}}: {{cashier.name}}
{{i18n.total}}: {{totals.total_display}}
```

As chaves comuns incluem `order`, `date`, `cashier`, `customer`, `item`, `sku`, `qty`, `unit_price`, `regular_price`, `selling_price`, `savings`, `discount`, `subtotal`, `total_saved`, `total`, `tax`, `paid`, `tendered`, `change`, `tax_summary`, `refunded`, `net_total`, `customer_note`, `thank_you_purchase`, `opening_hours` e as chaves de rótulo de ID fiscal, como `store_tax_id_label_eu_vat` e `customer_tax_id_label_other`. Chaves adicionais podem ser incluídas por extensões.

  </AccordionItem>
