Die Serverleistung wirkt sich direkt auf die Geschwindigkeit und Reaktionsfähigkeit Ihrer WCPOS-Anwendung aus. Diese Anleitung hilft Ihnen, die Serverleistung zu überwachen, zu diagnostizieren und zu optimieren, indem Sie die integrierten Metriken und Fehlersuche-Techniken nutzen.

## Verständnis der Servermetriken

WCPOS erhebt automatisch Serverleistungsmetriken mit jeder Datenabrufoperation (Produkte, Bestellungen, Kunden usw.). Sie können diese Metriken im [Protokoll](/support/logs)-Bildschirm einsehen.

### Typische Servermetriken

```json
{
  "total": "24692",
  "execution_time": "76.64 ms",
  "server_load": "[15.20605469,16.16357422,16.76806641]"
}
```

**Aufschlüsselung:**
- **`total`** - Anzahl der verarbeiteten Datensätze (24.692 Produkt-IDs)
- **`execution_time`** - Zeit, die benötigt wurde, um die Operation abzuschließen (76,64 Millisekunden)
- **`server_load`** - Serverlastdurchschnitt für 1, 5 und 15 Minuten

## Serverlast erklärt

Die Serverlast repräsentiert die durchschnittliche Systemlast über verschiedene Zeiträume:

- **Erster Wert** - 1-Minuten-Durchschnitt (15,21)
- **Zweiter Wert** - 5-Minuten-Durchschnitt (16,16)
- **Dritter Wert** - 15-Minuten-Durchschnitt (16,77)

### Lastinterpretation

Serverlastwerte können irreführend sein und sollten vorsichtig interpretiert werden:

:::warning Lastwerte können irreführend sein
Die Serverlastdurchschnittswerte korrelieren nicht immer direkt mit der Leistung. Ein Server mit hohen Lastwerten (15+) kann immer noch sehr reaktionsschnell sein, wenn er über ausreichende Ressourcen verfügt und gut optimiert ist. **Konzentrieren Sie sich auf die Ausführungszeiten anstatt nur auf die Lastwerte.**
:::

**Allgemeine Richtlinien:**
- **Last relativ zu CPU-Kernen** - Eine Last von 8,0 auf einem 8-Kern-Server bedeutet volle Auslastung
- **Eingehende vs. vorübergehende** - Kurze Spitzen sind normal, nachhaltig hohe Last kann auf Probleme hinweisen
- **Leistung ist wichtiger** - Ein reaktionsschneller Server mit hoher Last ist besser als ein langsamer Server mit niedriger Last

**Darauf sollten Sie achten:**
- **Ausführungszeiten, die über die Zeit zunehmen**
- **Last, die ohne Erklärung kontinuierlich wächst**
- **Hohe Last UND langsame Ausführungszeiten zusammen**

## Leistungsbenchmarks

### Richtlinien zur Ausführungszeit

| Operation | Gut | Akzeptabel | Schlecht | Kritisch |
|-----------|------|------------|------|----------|
| **Produktabruf** | < 100ms | 100-500ms | 500ms-2s | > 2s |
| **Bestell Erstellung** | < 200ms | 200-800ms | 800ms-3s | > 3s |

### Überlegungen zur Datensatzanzahl

Die Ausführungszeit sollte angemessen mit der Datensatzanzahl skalieren:

```json
// Good scaling examples
{"total": "100", "execution_time": "15.2 ms"}    // 0.15ms per record
{"total": "1000", "execution_time": "89.4 ms"}   // 0.09ms per record
{"total": "10000", "execution_time": "234.1 ms"} // 0.02ms per record

// Poor scaling examples
{"total": "100", "execution_time": "500.0 ms"}   // 5.0ms per record
{"total": "1000", "execution_time": "8000.0 ms"} // 8.0ms per record
```

## Diagnostizieren von Leistungsproblemen

### Schritt 1: Protokolle überwachen

1. Öffnen Sie **Protokolle** im Navigationsbereich
2. Führen Sie die langsame Operation durch (Produkte synchronisieren, Bestellung erstellen usw.)
3. Suchen Sie den entsprechenden Protokolleintrag
4. Erweitern Sie den Kontext, um die Metriken zu sehen

### Schritt 2: Analysieren der Metriken

**Hohe Ausführungszeit + Hohe Serverlast = Serverressourcenproblem**
```json
{
  "total": "5000",
  "execution_time": "3500.0 ms",
  "server_load": "[12.45, 11.23, 10.87]"
}
```
*Lösung: Serverressourcen erhöhen oder Serverkonfiguration optimieren*

**Hohe Ausführungszeit + Normale Serverlast = Plugin-/Datenbankproblem**
```json
{
  "total": "1000",
  "execution_time": "2800.0 ms",
  "server_load": "[1.23, 1.45, 1.67]"
}
```
*Lösung: Langsame Plugins identifizieren oder Datenbankabfragen optimieren*

**Normale Ausführungszeit + Hohe Serverlast = Allgemeine Serverüberlastung**
```json
{
  "total": "2000",
  "execution_time": "150.0 ms",
  "server_load": "[8.90, 9.12, 8.45]"
}
```
*Lösung: Serverlast von anderen Prozessen reduzieren oder Ressourcen aufrüsten*

## Häufige Leistungsprobleme

### 1. Unzureichende Serverressourcen

**Symptome:**
- Konsistent hohe Serverlast (> 4.0 auf den meisten Servern)
- Lange Ausführungszeiten bei allen Operationen
- Häufige Zeitüberschreitungen

**Lösungen:**
- **CPU aufrüsten** - Mehr Kerne bewältigen gleichzeitige Anfragen besser
- **RAM erhöhen** - Verringert die Festplatten-I/O und verbessert das Caching
- **SSD-Speicher verwenden** - Verbessert die Datenbankleistung erheblich
- **PHP-Einstellungen optimieren** - memory_limit, max_execution_time erhöhen

### 2. Langsame Datenbankabfragen

**Symptome:**
- Hohe Ausführungszeiten bei normaler Serverlast
- Besonders langsamer Abruf von Produk- oder Bestellinformationen
- Datenbankbezogene Fehlercodes in den Protokollen

**Lösungen:**
- **WooCommerce HPOS aktivieren** - Die größte Leistungssteigerung der Datenbank
- **Objekt-Cache verwenden** - Redis oder Memcached, sofern verfügbar von Ihrem Hoster
- **WordPress aktuell halten** - Kernaktualisierungen enthalten oft Datenbankoptimierungen
- **Revisionen von Beiträgen begrenzen** - Fügen Sie `define('WP_POST_REVISIONS', 3);` zu wp-config.php hinzu

### 3. Plugin-Interferenzen

**Symptome:**
- Plötzliche Leistungseinbußen nach Plugin-Updates
- Bestimmte Operationen deutlich langsamer als andere
- Hohe Ausführungszeiten bei normaler Serverlast

**Fehlerbehebung:**
1. **Im Staging testen** - Deaktivieren Sie alle Plugins außer WooCommerce und WCPOS
2. **Basislinien messen** - Ausführungszeiten mit minimalen Plugins aufzeichnen
3. **Allmählich aktivieren** - Fügen Sie Plugins nacheinander hinzu, um Übeltäter zu identifizieren
4. **Plugins-Hooks überprüfen** - Suchen Sie nach Plugins, die in WooCommerce-Aktionen eingreifen

**Häufig problematische Plugins:**
- Schwere SEO-Plugins während der Produktoperationen
- Komplexe Bestandsmanagementsysteme
- Echtzeit-Analyse-/Tracking-Plugins
- Schlecht codierte benutzerdefinierte Plugins

### 4. WordPress/WooCommerce-Konfiguration

**Symptome:**
- Inkonsistente Leistung
- Speicherbezogene Fehler in Protokollen
- Langsame Admin-Oberfläche

**Optimierungscheckliste:**
- **PHP-Version** - Verwenden Sie PHP 8.0+ für bessere Leistung
- **WooCommerce HPOS** - Aktivieren Sie die Hochleistungs-Bestellstorage (siehe unten)
- **WordPress-Caching** - Aktivieren Sie das Objekt-Caching, wenn verfügbar
- **WooCommerce-Einstellungen** - Optimieren Sie die Produktbildgrößen

### WooCommerce Hochleistungs-Bestellstorage (HPOS)

:::tip Größter Leistungsgewinn
**HPOS ist eine der bedeutendsten Leistungsverbesserungen, die Sie für WooCommerce vornehmen können.** Es speichert Bestellungen in benutzerdefinierten Datenbanktabellen anstelle der WordPress-Posts-Tabelle, was die Leistung für Geschäfte mit vielen Bestellungen erheblich verbessert.
:::

**Vorteile:**
- **Schnellere Bestellabfragen** - Bestellungen werden in optimierter Datenbankstruktur gespeichert
- **Verringerten Datenbanklast** - Trennt Bestellungen von Beiträgen/Seiten
- **Bessere Skalierbarkeit** - Bewältigt große Bestellzahlen effizient
- **Verbesserte Leistungen im Admin-Bereich** - Schnellere Bildschirme zur Bestellverwaltung

**Aktivierung:**
1. Gehen Sie zu `WooCommerce > Einstellungen > Erweitert > Funktionen`
2. Aktivieren Sie "Hochleistungs-Bestellspeicherung"
3. Folgen Sie dem Migrationsprozess

**Erfahren Sie mehr:**
- [WooCommerce HPOS-Dokumentation](https://woocommerce.com/document/high-performance-order-storage/)
- [HPOS-Migrationsleitfaden](https://woocommerce.com/document/high-performance-order-storage/#section-3)

## Beste Praktiken zur Serverüberwachung

### 1. Regelmäßige Leistungsüberprüfungen

- **Wöchentliche Überprüfungen** - Überprüfen Sie Protokolle auf Leistungstrends
- **Basislinienmessungen** - Normale Ausführungszeiten aufzeichnen
- **Spitzenzeitüberwachung** - Während hochfrequentierter Zeiten überwachen

### 2. Leistungswarnungen festlegen

Überwachen Sie diese Warnzeichen:
- Ausführungszeiten > 1000ms konsistent
- Serverlast > 5.0 über längere Zeiträume
- Häufige Zeitüberschreitungsfehler in den Protokollen

### 3. Kapazitätsplanung

Wachstumstrends verfolgen:
- **Wachstum der Datensatzanzahl** - Produkte, Bestellungen, Kunden
- **Leistungsabfall** - Wie die Ausführungszeit skaliert
- **Ressourcennutzung** - CPU, Speicher, Festplattennutzung

## Strategien zur Serveroptimierung

### 1. Best Practices für WordPress/WooCommerce

**HPOS aktivieren:**
- Der mit Abstand wirkungsvollste Leistungsverbesserung für WooCommerce
- Weitere Informationen im obigen HPOS-Abschnitt

**PHP-Konfiguration (fragen Sie Ihren Hoster):**
```ini
memory_limit = 512M
max_execution_time = 300
max_input_vars = 3000
```

**WordPress-Konfiguration:**
```php
// In wp-config.php - limit post revisions
define('WP_POST_REVISIONS', 3);

// Enable WordPress debug logging if needed
define('WP_DEBUG_LOG', true);
```

### 2. Hosting-Level-Optimierungen

**Objekt-Caching:**
- Fragen Sie Ihren Hoster nach der Verfügbarkeit von Redis oder Memcached
- Viele verwaltete WordPress-Hosts bieten dies automatisch an

**PHP-Version:**
- Verwenden Sie PHP 8.0+ für erhebliche Leistungsverbesserungen
- Die meisten Hoster ermöglichen den einfachen Wechsel der PHP-Version

**Serverressourcen:**
- Stellen Sie sicher, dass ausreichend RAM vorhanden ist (mindestens 1 GB, vorzugsweise > 2 GB)
- SSD-Speicher bietet eine viel bessere Datenbankleistung als herkömmliche Laufwerke

## Wann Sie Hilfe suchen sollten

Kontaktieren Sie Ihren Hosting-Anbieter oder einen WordPress-Entwickler, wenn:

- **Serverlast konsistent > 8.0** trotz Optimierungsversuche
- **Ausführungszeiten > 5000ms** für einfache Operationen
- **Speicherfehler** häufig in Protokollen auftreten
- **Datenbankabfragen dauern > 2 Sekunden** konsistent

Stellen Sie ihnen zur Verfügung:
- Servermetriken aus Ihren Protokollen
- Liste der aktiven Plugins
- Serverspezifikationen (CPU, RAM, Speichertyp)
- WordPress- und WooCommerce-Versionen
