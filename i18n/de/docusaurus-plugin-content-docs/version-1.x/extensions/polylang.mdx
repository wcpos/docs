DE: # WCPOS Polylang

DE: Adds [Polylang](https://polylang.pro/) awareness to WCPOS so the POS only shows products for a single language — no duplicated translations in product search, the catalog grid, or cashier workflows. WCPOS Pro stores can pin a per-store language; free installs fall back to the Polylang default language.

## Ubersetzt: What It Does {#what-it-does}

- DE: Filters WCPOS product and variation REST queries by language.
- DE: Intercepts WCPOS **fast-sync** routes (the lightweight `posts_per_page=-1` + `fields` requests the POS uses to refresh its local index) so translated duplicates never reach the client.
- DE: On free installs, applies the Polylang default language.
- DE: On Pro installs, each store can choose its own language from a new **Language** section in the store editor.
- DE: Respects WCPOS **POS-only** product visibility when building the fast-sync payload.

DE: The integration no-ops cleanly when Polylang is not active — you can install the plugin ahead of enabling Polylang without errors.

## Ubersetzt: Installation {#installation}

<Steps>
  <StepCard number="1" title="DE: Install Polylang">

DE: Install [Polylang](https://wordpress.org/plugins/polylang/) (or Polylang Pro) and configure your site languages as normal. Make sure at least one language is set as the default.

  </StepCard>
  <StepCard number="2" title="DE: Install WCPOS Polylang">

DE: Install from the WCPOS extensions directory at `WP Admin > POS > Settings > Extensions`, or download the latest release from the [GitHub releases page](https://github.com/wcpos/wcpos-polylang/releases) and upload via `Plugins > Add New > Upload Plugin`.

  </StepCard>
  <StepCard number="3" title="DE: (Pro) Set a per-store language" isLast>

DE: If you run [multiple stores](/stores) on WCPOS Pro, go to `POS > Stores`, edit a store, and pick its language from the **Language** sidebar section. Leave it at *Default* to use your Polylang default language.

  </StepCard>
</Steps>

## Ubersetzt: Per-Store Language (Pro) {#per-store-language-pro}

DE: On WCPOS Pro, the plugin adds a **Language** section to the store editor sidebar. Each store can be pinned to a single Polylang language slug — products served to that store are filtered to that language only. Stores left on *Default* use the Polylang default language.

DE: The per-store value is saved against the store post as `_wcpos_polylang_language` meta and is exposed via the WCPOS Pro stores REST API (`/wcpos/v1/stores`), so it round-trips through the POS like any other store setting.

## Ubersetzt: Compatibility Notes {#compatibility-notes}

- DE: **POS-only products:** when POS-only mode is enabled in WCPOS settings, online-only product IDs are excluded from the fast-sync payload so they do not leak into the POS.
- DE: **Free installs:** there is no UI for changing the language per store — the plugin uses Polylang's default language. If you need per-store languages, upgrade to [WCPOS Pro](/getting-started/pro-license).
- DE: **Plugin unavailable:** if Polylang is deactivated, the plugin silently does nothing. It will not throw errors or block the POS.

## Ubersetzt: Developer Hooks {#developer-hooks}

DE: For advanced use, the plugin exposes a few filters:

| DE: Filter | DE: Purpose | DE:
| --- | --- |
| DE: `wcpos_polylang_resolved_language` | DE: Override the language used for a given request. Receives the resolved slug and the `WP_REST_Request`. | DE:
| DE: `wcpos_polylang_default_language` | DE: Override the fallback language when no per-store value is set. | DE:
| DE: `wcpos_polylang_is_supported` | DE: Force the plugin on or off regardless of Polylang detection. | DE:
| DE: `wcpos_polylang_minimum_version` | DE: Require a minimum Polylang version (default: no version gate). | DE:
