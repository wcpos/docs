IT: # WCPOS Polylang

IT: Adds [Polylang](https://polylang.pro/) awareness to WCPOS so the POS only shows products for a single language — no duplicated translations in product search, the catalog grid, or cashier workflows. WCPOS Pro stores can pin a per-store language; free installs fall back to the Polylang default language.

## Tradotto: What It Does {#what-it-does}

- IT: Filters WCPOS product and variation REST queries by language.
- IT: Intercepts WCPOS **fast-sync** routes (the lightweight `posts_per_page=-1` + `fields` requests the POS uses to refresh its local index) so translated duplicates never reach the client.
- IT: On free installs, applies the Polylang default language.
- IT: On Pro installs, each store can choose its own language from a new **Language** section in the store editor.
- IT: Respects WCPOS **POS-only** product visibility when building the fast-sync payload.

IT: The integration no-ops cleanly when Polylang is not active — you can install the plugin ahead of enabling Polylang without errors.

## Tradotto: Installation {#installation}

<Steps>
  <StepCard number="1" title="IT: Install Polylang">

IT: Install [Polylang](https://wordpress.org/plugins/polylang/) (or Polylang Pro) and configure your site languages as normal. Make sure at least one language is set as the default.

  </StepCard>
  <StepCard number="2" title="IT: Install WCPOS Polylang">

IT: Install from the WCPOS extensions directory at `WP Admin > POS > Settings > Extensions`, or download the latest release from the [GitHub releases page](https://github.com/wcpos/wcpos-polylang/releases) and upload via `Plugins > Add New > Upload Plugin`.

  </StepCard>
  <StepCard number="3" title="IT: (Pro) Set a per-store language" isLast>

IT: If you run [multiple stores](/stores) on WCPOS Pro, go to `POS > Stores`, edit a store, and pick its language from the **Language** sidebar section. Leave it at *Default* to use your Polylang default language.

  </StepCard>
</Steps>

## Tradotto: Per-Store Language (Pro) {#per-store-language-pro}

IT: On WCPOS Pro, the plugin adds a **Language** section to the store editor sidebar. Each store can be pinned to a single Polylang language slug — products served to that store are filtered to that language only. Stores left on *Default* use the Polylang default language.

IT: The per-store value is saved against the store post as `_wcpos_polylang_language` meta and is exposed via the WCPOS Pro stores REST API (`/wcpos/v1/stores`), so it round-trips through the POS like any other store setting.

## Tradotto: Compatibility Notes {#compatibility-notes}

- IT: **POS-only products:** when POS-only mode is enabled in WCPOS settings, online-only product IDs are excluded from the fast-sync payload so they do not leak into the POS.
- IT: **Free installs:** there is no UI for changing the language per store — the plugin uses Polylang's default language. If you need per-store languages, upgrade to [WCPOS Pro](/getting-started/pro-license).
- IT: **Plugin unavailable:** if Polylang is deactivated, the plugin silently does nothing. It will not throw errors or block the POS.

## Tradotto: Developer Hooks {#developer-hooks}

IT: For advanced use, the plugin exposes a few filters:

| IT: Filter | IT: Purpose | IT:
| --- | --- |
| IT: `wcpos_polylang_resolved_language` | IT: Override the language used for a given request. Receives the resolved slug and the `WP_REST_Request`. | IT:
| IT: `wcpos_polylang_default_language` | IT: Override the fallback language when no per-store value is set. | IT:
| IT: `wcpos_polylang_is_supported` | IT: Force the plugin on or off regardless of Polylang detection. | IT:
| IT: `wcpos_polylang_minimum_version` | IT: Require a minimum Polylang version (default: no version gate). | IT:
