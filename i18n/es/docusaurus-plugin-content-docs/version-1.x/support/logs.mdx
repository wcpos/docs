El POS tiene dos interfaces de registros: la **pantalla Registros de la aplicación** (dentro de la aplicación POS) y la **página Registros de WP Admin** (dentro de WordPress). Juntas ofrecen una visión completa de la actividad del POS para depuración y auditoría.

## Registros de la aplicación POS {#pos-application-logs}

### Acceso a los registros {#accessing-logs}

Abra la pantalla Registros desde el panel de navegación izquierdo (el icono de latido).

### Resumen de la interfaz {#interface-overview}

#### Buscar y filtrar {#search--filter}

- **Buscar registros** - Busque entradas de registro específicas por el contenido del mensaje
- **Filtro de nivel de registro** - Filtre por nivel de registro (info, warn, error)
- **Configuración de visualización** - Configurar columnas visibles

#### Tabla de registros {#log-table}

Cada entrada de registro muestra:

| Columna | Descripción |
|--------|-------------|
| **Hora** | Cuándo ocurrió el evento |
| **Nivel** | Gravedad: información (gris), advertencia (naranja), error (rojo) |
| **Mensaje** | Descripción legible del evento |
| **Contexto** | JSON expandible con detalles adicionales |
| **Código** | Código de error (enlaza a la documentación) |

### Niveles de registro {#log-levels}

#### Información (gris) {#info-grey}

Operaciones normales y eventos correctos:
- "Se obtuvieron todos los ID de productos"
- "Se sincronizaron los clientes nuevos"
- "Se obtuvieron todos los ID de impuestos"

#### Advertencia (naranja) {#warning-orange}

Problemas no críticos que pueden requerir atención:
- "No se pudo actualizar la sesión"
- "Intento de reintentar la sincronización"

#### Error (rojo) {#error-red}

Problemas que deben solucionarse:
- "Lo sentimos, no puede listar recursos" (problema de permisos)
- "Error en la solicitud API"
- "Error de sincronización de la base de datos"

#### Auditoría (azul) {#audit-blue}

El nivel de registro de auditoría registra los cambios en la base de datos realizados por el POS, capturando el estado completo anterior y posterior de cada modificación. Esto es útil para hacer un seguimiento exacto de qué cambió y cuándo, por ejemplo durante la creación de pedidos, los ajustes de inventario o las actualizaciones de clientes.

Las entradas de auditoría incluyen:
- **Tipo de entidad** - El recurso que se modificó (p. ej., pedido, producto, cliente)
- **Acción** - El tipo de cambio (crear, actualizar, eliminar)
- **Estado anterior** - Los valores anteriores al cambio
- **Estado posterior** - Los nuevos valores después del cambio

### Datos de contexto {#context-data}

Seleccione la flecha de expansión en cualquier entrada de registro para ver el contexto JSON detallado:

```json
{
  "total": "6",
  "execution_time": "54.79 ms",
  "server_load": "[10.51, 10.42, 9.98]"
}
```

Estos datos incluyen:
- **Recuento de elementos** - Cuántos registros se vieron afectados
- **Tiempo de ejecución** - Cuánto tardó la operación
- **Carga del servidor** - Métricas de rendimiento del servidor
- **Detalles de solicitud/respuesta** - Para depurar problemas de API

### Códigos de error {#error-codes}

Muchas entradas del registro incluyen un **código de error** (p. ej., `API02004`, `DB01001`). Estos códigos enlazan directamente con la documentación de [Códigos de error](/error-codes), que proporciona:

- Explicación detallada del error
- Causas habituales
- Pasos para solucionar problemas

### Entradas habituales del registro {#common-log-entries}

#### Operaciones de sincronización {#sync-operations}

```
[info] Fetched all IDs for products    {...} 3 items
[info] Synced new customers            {...} 1 item
[info] Fetched all IDs for taxes       {...} 3 items
```

Estas indican actividad normal de sincronización. El contexto muestra cuántos elementos se sincronizaron.

#### Errores de permisos {#permission-errors}

```
[error] Sorry, you cannot list resources    {...} API02004
```

Esto suele significar que el usuario no tiene permiso para acceder a un recurso. Compruebe los roles de usuario en `WP Admin > POS > Ajustes > Access`.

#### Problemas de sesión {#session-issues}

```
[warn] Unable to refresh session    {...} API02007
```

El POS no pudo actualizar la sesión de autenticación. Puede ser necesario cerrar sesión y volver a iniciarla.

## Página de registros de WP Admin {#wp-admin-logs-page}

El plugin POS también proporciona una página de registros del lado del servidor en el área de administración de WordPress. Esta página muestra advertencias y errores específicos de POS registrados en el servidor, lo que puede ayudar a diagnosticar problemas que quizá no aparezcan en la propia aplicación POS.

### Acceso a la página de registros de WP Admin {#accessing-the-wp-admin-logs-page}

Acceda a **WP Admin > POS > Ajustes**, seleccione la pestaña **Herramientas** y haga clic en **Registros**.

### Insignia de no leídos {#unread-badge}

El elemento de menú Registros muestra una insignia que indica las entradas de registro no leídas:

- **Insignia roja** - Hay entradas no leídas de nivel de error
- **Insignia ámbar** - Hay entradas no leídas de nivel de advertencia (pero no errores)

Al visitar la página Registros se restablecen los recuentos de no leídos. Los recuentos se registran por usuario, por lo que cada administrador ve su propio estado de no leídos.

### Filtrado de entradas {#filtering-entries}

Use el filtro de nivel para limitar las entradas de registro:

- **Todas** - Mostrar todas las entradas de registro
- **Errores** - Mostrar solo las entradas de nivel de error
- **Advertencias** - Mostrar solo las entradas de nivel de advertencia

### Entradas expandibles {#expandable-entries}

Los mensajes de registro largos se truncan aproximadamente a los 100 caracteres. Seleccione una entrada para expandirla y ver el mensaje completo junto con cualquier dato de contexto adicional.

### Paginación {#pagination}

Las entradas de registro se paginan en el servidor. Use los controles de paginación para navegar por entradas anteriores.

### Banner de errores fatales {#fatal-errors-banner}

Si el servidor tiene archivos `fatal-errors-*.log`, la página Registros muestra un banner de advertencia en la parte superior. Este banner enlaza a **WooCommerce > Estado > Registros**, donde se pueden ver los registros completos de errores fatales. Los errores fatales suelen ser fallos de nivel PHP que impiden que la API de POS responda.

## Uso de los registros para soporte {#using-logs-for-support}

Al solicitar soporte:

1. **Reproducir el problema** - Realizar la acción que causa el problema
2. **Comprobar los registros** - Buscar errores (rojo) o advertencias (naranja)
3. **Expandir el contexto** - Obtener los detalles completos en JSON
4. **Anotar el código de error** - Incluirlo en la solicitud de soporte
5. **Copiar las entradas relevantes** - Compartir los mensajes del registro con contexto

## Supervisión del rendimiento {#performance-monitoring}

Los registros incluyen métricas de tiempo de ejecución y carga del servidor. Si observa:

- **Tiempos de ejecución altos** - Es posible que el servidor esté sobrecargado o lento
- **Valores altos de carga del servidor** - Considere optimizar el servidor
- **Tiempos de espera frecuentes** - Comprobar la conectividad de red
