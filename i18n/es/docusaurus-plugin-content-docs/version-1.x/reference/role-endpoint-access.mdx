Esta página documenta qué endpoints REST de WCPOS son accesibles para cada uno de los tres roles predeterminados utilizados en los flujos de trabajo del POS:

- `administrator`
- `shop_manager`
- `cashier`

Para la configuración de roles y capacidades orientada al usuario, consulte [Acceso al POS](/settings/wp-admin/access).

## Resumen {#summary}

El modelo de acceso predeterminado es:

- **`administrator`** — acceso completo al POS y a la administración.
- **`shop_manager`** — acceso completo al POS y a la administración, excepto las acciones de extensiones de plugins que requieren capacidades para instalar plugins.
- **`cashier`** — acceso al flujo de venta normal de POS, pero sin acceso a gestión, ajustes, registros ni administración de tiendas.

La matriz de endpoints se verifica mediante pruebas de integración que ejercitan la misma ruta de autenticación que la aplicación POS (tokens de acceso bearer de WCPOS, no solo `wp_set_current_user()`). Las aserciones de endpoints permitidos confirman que un rol no recibe `401` ni `403`. Algunos endpoints aún pueden devolver datos normales o errores de negocio como `400` o `404` cuando la prueba usa fixtures mínimos; esas respuestas siguen demostrando que el usuario superó la capa de autenticación/permisos.

## Cobertura de caducidad de tokens {#token-expiry-coverage}

| Escenario | Administrador | Gestor de la tienda | Cajero | Resultado esperado |
|---|---:|---:|---:|---|
| Token de acceso válido | ✅ probado | ✅ probado | ✅ probado | Los endpoints permitidos no devuelven `401` ni `403` |
| Token de acceso caducado | ✅ probado | ✅ probado | ✅ probado | La solicitud falla en el control de autenticación de WCPOS con el usuario `0` |
| Token de actualización válido después de un acceso caducado | ✅ probado | ✅ probado | ✅ probado | Se emite un nuevo token de acceso y se restaura el acceso |
| Token de actualización caducado | ✅ probado | ✅ probado | ✅ probado | No se puede emitir un nuevo token de acceso |

## Diagnóstico de fallos {#diagnosing-failures}

Si un `shop_manager` predeterminado ve un error como:

> La solicitud sigue sin estar autorizada después de actualizar el token; inicie sesión de nuevo

La matriz de endpoints indica que esto **no** se debe a que el rol `shop_manager` predeterminado carezca fundamentalmente de acceso a POS. Posibles causas que conviene investigar:

1. Corrupción de roles o capacidades específica del sitio.
2. Un endpoint no cubierto por la matriz.
3. Comportamiento de reintento con token de acceso obsoleto o caducado en el cliente.
4. Interferencia de plugins de seguridad o autenticación de terceros.
5. Reglas de acceso Pro o específicas de la tienda, o código personalizado.

### Fallo de permisos frente a fallo de token {#permission-failure-vs-token-failure}

El frontend de POS puede mostrar el mismo error de actualización de token en ambos casos después de recibir un `401` o `403`, intentar actualizar el token y que el reintento siga sin autorización. Los dos modos de fallo se ven diferentes en el lado del servidor:

Un **fallo real de rol/capacidad** suele verse así:

```text
status: 403
current_user: <actual user id>
access token expired: false
```

Un **fallo de token/autenticación** suele verse así:

```text
status: 403
current_user: 0
access token expired: true or invalid
```

Use esta distinción al clasificar problemas de acceso a POS: un `current_user` distinto de cero apunta a las capacidades, mientras que `current_user: 0` apunta a la capa de autenticación/token.
