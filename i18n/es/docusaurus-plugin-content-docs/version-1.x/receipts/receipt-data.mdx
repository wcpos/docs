Las plantillas HTML sin lógica de WCPOS y las plantillas XML térmicas se generan a partir de la misma carga útil canónica de recibos. Utilice rutas con puntos de Mustache como `{{order.number}}`, `{{store.name}}` y `{{totals.total_display}}`. Los arreglos se generan mediante secciones:

```mustache
{{#lines}}
  {{name}} x {{qty}} — {{line_total_display}}
{{/lines}}
```

El contrato canónico lo genera el constructor de datos de recibos de WCPOS en el servidor y se replica en el renderizador de recibos sin conexión de la aplicación. Los recibos se abren de inmediato a partir de datos locales y luego se actualizan con la respuesta del servidor cuando está disponible, por lo que las plantillas personalizadas deben usar los campos siguientes en lugar de los métodos de pedidos de PHP.

Datos de recibos v1.1 añade campos registrados de precio normal, precio de venta antes de aplicar cupones y ahorro para cada línea de producto. También proporciona totales de ahorro a nivel de pedido, incluido un valor de **Total ahorrado** que combina el ahorro del precio normal con los descuentos de WooCommerce sin modificar los totales ni los informes del pedido.

## Reglas de renderizado {#rendering-rules}

### Campos de moneda {#currency-fields}

Los campos monetarios numéricos se conservan como números y el renderizador añade campos `_display` adaptados a la configuración regional para la salida de la plantilla:

| Campo numérico | Campo de visualización |
|---------------|---------------|
| `totals.total` | `totals.total_display` |
| `lines[].line_total` | `lines[].line_total_display` |
| `payments[].amount` | `payments[].amount_display` |
| `tax_summary[].tax_amount` | `tax_summary[].tax_amount_display` |

Al imprimir recibos, se deben priorizar los campos `_display`. Los campos numéricos deben usarse únicamente para secciones condicionales o salidas legibles por máquina.

### Campos adaptados a la visualización de impuestos {#tax-display-aware-fields}

Varios campos tienen variantes con impuestos incluidos y excluidos, además de un valor auxiliar para la visualización. Este valor sigue la configuración de visualización de impuestos del carrito de la tienda.

| Campo auxiliar | Campo con impuestos incluidos | Campo con impuestos excluidos |
|-------------------|-----------------|-----------------|
| `lines[].regular_price` | `lines[].regular_price_incl` | `lines[].regular_price_excl` |
| `lines[].selling_price` | `lines[].selling_price_incl` | `lines[].selling_price_excl` |
| `lines[].unit_savings` | `lines[].unit_savings_incl` | `lines[].unit_savings_excl` |
| `lines[].line_regular_total` | `lines[].line_regular_total_incl` | `lines[].line_regular_total_excl` |
| `lines[].line_selling_total` | `lines[].line_selling_total_incl` | `lines[].line_selling_total_excl` |
| `lines[].line_savings` | `lines[].line_savings_incl` | `lines[].line_savings_excl` |
| `lines[].unit_price` | `lines[].unit_price_incl` | `lines[].unit_price_excl` |
| `lines[].unit_subtotal` | `lines[].unit_subtotal_incl` | `lines[].unit_subtotal_excl` |
| `lines[].line_subtotal` | `lines[].line_subtotal_incl` | `lines[].line_subtotal_excl` |
| `lines[].discounts` | `lines[].discounts_incl` | `lines[].discounts_excl` |
| `lines[].line_total` | `lines[].line_total_incl` | `lines[].line_total_excl` |
| `fees[].total` | `fees[].total_incl` | `fees[].total_excl` |
| `shipping[].total` | `shipping[].total_incl` | `shipping[].total_excl` |
| `discounts[].total` | `discounts[].total_incl` | `discounts[].total_excl` |
| `totals.subtotal` | `totals.subtotal_incl` | `totals.subtotal_excl` |
| `totals.discount_total` | `totals.discount_total_incl` | `totals.discount_total_excl` |
| `totals.sale_savings_total` | `totals.sale_savings_total_incl` | `totals.sale_savings_total_excl` |
| `totals.total_saved` | `totals.total_saved_incl` | `totals.total_saved_excl` |
| `totals.total` | `totals.total_incl` | `totals.total_excl` |

### Objetos de fecha {#date-objects}

Los campos de fecha son objetos con varias variantes preformateadas. Esto evita tener que dar formato a las fechas dentro de Mustache.

| Campo | Descripción |
|-------|-------------|
| `datetime`, `date`, `time` | Cadenas de fecha/hora predeterminadas |
| `datetime_short`, `datetime_long`, `datetime_full` | Formatos combinados adaptados a la configuración regional |
| `date_short`, `date_long`, `date_full` | Formatos solo de fecha adaptados a la configuración regional |
| `date_ymd`, `date_dmy`, `date_mdy` | Formatos de fecha con orden fijo |
| `weekday_short`, `weekday_long` | Nombres de los días |
| `day`, `month`, `month_short`, `month_long`, `year` | Componentes individuales de la fecha |

Objetos de fecha disponibles: `order.created`, `order.paid`, `order.completed`, `order.printed` y `refunds[].date`. `order.printed` se actualiza en el momento de la generación, lo que resulta útil para las reimpresiones.

## Relación entre los precios {#price-model}

Una línea de producto contiene varios precios, y los nombres se entienden mejor al verlos sobre el importe. Esta es la línea de espresso del pedido de muestra: un producto con un precio normal de 17,00 $, en oferta por 14,50 $, más un cupón del 10 %.

El ahorro y los descuentos de cupones son conceptos distintos: el ahorro compara el precio de venta con el precio normal registrado, mientras que `discounts` representa el cálculo de cupones de WooCommerce. Los totales e informes de pedidos de WooCommerce solo contienen la parte correspondiente al cupón. Por eso `total_saved` es la cifra única para el cliente que combina ambos importes.

### líneas {#lines}

Itere con `{{#lines}}...{{/lines}}`.

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `key` | cadena | Clave estable de la línea/ID del artículo del pedido |
| `sku` | cadena | SKU del producto |
| `name` | cadena | Nombre mostrado del producto o de la línea |
| `qty` | número | Cantidad vendida |
| `qty_refunded` | número | Cantidad reembolsada de esta línea |
| `regular_price` / `_incl` / `_excl` | número/nulo | Precio unitario normal registrado |
| `selling_price` / `_incl` / `_excl` | número/null | Precio unitario de venta antes del cupón — el precio de POS registrado cuando está disponible; de lo contrario, se obtiene del subtotal de línea almacenado |
| `unit_savings` / `_incl` / `_excl` | número/nulo | `regular_price - selling_price`, con un mínimo de cero |
| `line_regular_total` / `_incl` / `_excl` | número/nulo | Precio normal registrado multiplicado por la cantidad |
| `line_selling_total` / `_incl` / `_excl` | número/nulo | Precio de venta antes de los cupones multiplicado por la cantidad |
| `line_savings` / `_incl` / `_excl` | número/nulo | Ahorro total entre el precio normal y el precio de venta de esta línea |
| `savings_in_discounts` | booleano | Verdadero cuando un pedido heredado ya incluye el mismo ahorro en `discounts` |
| `unit_subtotal` / `_incl` / `_excl` | número | Valor unitario de WooCommerce antes del descuento; no lo use como precio normal |
| `unit_price` / `_incl` / `_excl` | número | Valor unitario final después de los descuentos de WooCommerce |
| `line_subtotal` / `_incl` / `_excl` | número | Subtotal de la línea de WooCommerce antes del descuento |
| `discounts` / `_incl` / `_excl` | número | Descuento de línea nativo de WooCommerce como valor positivo; normalmente solo de cupones en los pedidos actuales |
| `line_total` / `_incl` / `_excl` | número | Total final de la línea |
| `total_refunded` | número | Total reembolsado para esta línea como valor positivo |
| `taxes` | matriz | Filas de impuestos por tasa para esta línea |
| `meta` | matriz | Metadatos del artículo del pedido como pares `{key, value}` |
| `attributes` | matriz | Atributos del producto o la variación como pares `{key, value}` |

Cada campo monetario tiene un complemento `_display` con formato, incluidos `regular_price_display`, `selling_price_display`, `unit_savings_display`, `line_regular_total_display`, `line_selling_total_display` y `line_savings_display`. Los campos inclusivos y exclusivos también tienen variantes `_display`. Cuando un campo de ahorro que admite valores nulos es `null`, su complemento `_display` se omite por completo.

Los campos de ahorro son `null` cuando el pedido no contiene suficientes datos registrados de precios de POS. Por lo tanto, las plantillas deben incluirlos en secciones Mustache en lugar de asumir que todos los pedidos históricos tienen un precio normal.

### Mostrar el precio habitual y el ahorro {#displaying-regular-price-and-savings}

Para un diseño con precio unitario, muestre el precio habitual registrado, el precio de venta y el ahorro de esta manera:

```mustache
{{#lines}}
  {{#unit_savings}}
    <span style="text-decoration: line-through;">{{regular_price_display}}</span>
    <span>{{selling_price_display}}</span>
    <span>{{i18n.savings}}: -{{unit_savings_display}}</span>
  {{/unit_savings}}
{{/lines}}
```

Utilice los campos de total de línea cuando la cantidad sea importante:

```mustache
{{#lines}}
  {{#line_savings}}
    <span style="text-decoration: line-through;">{{line_regular_total_display}}</span>
    <span>{{i18n.savings}}: -{{line_savings_display}}</span>
  {{/line_savings}}
{{/lines}}
```

`selling_price` es el precio antes de los cupones. Utilice `unit_price` o `line_total` para el importe final después de los descuentos de WooCommerce.

:::note Reimpresión de pedidos creados antes de la v1.9.0
Los pedidos antiguos de WCPOS pueden incluir ya la reducción entre el precio habitual y el precio de venta en `discounts`. Si la plantilla personalizada imprime tanto los ahorros de línea como los descuentos de línea, utilice `savings_in_discounts` para evitar mostrar la misma reducción dos veces:

```mustache
{{#lines}}
  {{#line_savings}}
    <span style="text-decoration: line-through;">{{line_regular_total_display}}</span>
    {{^savings_in_discounts}}
      <span>{{i18n.savings}}: -{{line_savings_display}}</span>
    {{/savings_in_discounts}}
  {{/line_savings}}
  {{#discounts}}
```

  <AccordionItem question="totales — totales de pedidos, pagos, reembolsos y artículos">

### totales {#totals}

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `totals.subtotal` / `_incl` / `_excl` | número | Subtotal del pedido antes de los descuentos |
| `totals.discount_total` / `_incl` / `_excl` | número | Total de descuentos del pedido como valor positivo |
| `totals.sale_savings_total` / `_incl` / `_excl` | número/null | Ahorro total registrado entre el precio habitual y el precio de venta en todas las líneas de productos, incluidas las ofertas del catálogo y los cambios de precio en caja |
| `totals.total_saved` / `_incl` / `_excl` | número/null | Descuentos de WooCommerce y ahorros sobre el precio habitual combinados, sin la superposición heredada |
| `totals.total_saved_complete` | booleano | Indica si el recibo tiene suficientes datos de precios registrados para calcular los totales de ahorro completos |
| `totals.tax_total` | número | Importe total de impuestos |
| `totals.total` / `_incl` / `_excl` | número | Total general del pedido |
| `totals.paid_total` | número | Importe pagado/aplicado |
| `totals.change_total` | número | Cambio devuelto al cliente |
| `totals.refund_total` | número | Total reembolsado como valor positivo |
| `totals.net_total` | número | `total - refund_total`, con un mínimo de cero |
| `totals.total_qty` | número | Suma de las cantidades de las líneas de artículos |
| `totals.line_count` | número | Número de filas de líneas de productos |

Las variantes formateadas incluyen `subtotal_display`, `discount_total_display`, `sale_savings_total_display`, `total_saved_display`, `tax_total_display`, `total_display`, `paid_total_display`, `change_total_display`, `refund_total_display` y `net_total_display`, además de variantes inclusivas/exclusivas cuando corresponda.

Utilice `total_saved` cuando se necesite una única cifra visible para el cliente que refleje todo lo ahorrado. No lo calcule sumando `discount_total` y `sale_savings_total` en la plantilla: los pedidos creados antes de la v1.9.0 pueden incluir ya el ahorro de precio de POS en el total de descuentos de WooCommerce. El campo canónico `total_saved` detecta y elimina esa superposición.

Los totales de ahorro son `null` y `total_saved_complete` es false cuando alguna línea de producto no cuenta con suficientes datos de precio registrados para obtener un total fiable. Proteja la fila con `total_saved_complete` y use después `total_saved` para ocultar un valor cero:

```mustache
{{#totals.total_saved_complete}}
  {{#totals.total_saved}}
    <span>{{i18n.total_saved}}: {{totals.total_saved_display}}</span>
  {{/totals.total_saved}}
{{/totals.total_saved_complete}}
```

Las plantillas incluidas con precios usan la misma estructura de protección con las variantes con impuestos incluidos (`total_saved_incl` y `total_saved_incl_display`), y muestran **Total ahorrado** solo cuando el valor completo es mayor que cero.

  </AccordionItem>
