Receipt templates control what your customers see when you print or display a receipt from the POS. You can write them in PHP (full WordPress/WooCommerce access) or use a simple logicless syntax with `{{variable}}` placeholders.

:::tip Offline Printing
Logicless templates can render entirely on the device — no server connection required. If your internet drops mid-shift, receipts still print. For retail environments where network reliability is a concern, this matters a lot. See [Choosing an Engine](#choosing-an-engine) below.
:::

## Quick Start

The fastest way to get going is to install one of the **Starter Templates** that ship with the plugin:

1. Go to **WP Admin > POS > Templates**
2. Find the **Starter Templates** section
3. Click **Install** on the one that fits your use case
4. Edit the template in the code editor, then click **Activate** when you're happy

Three starters are included out of the box:

| Starter | What it demonstrates |
|---------|---------------------|
| **Minimal Receipt** | Clean layout using `$receipt_data` — good starting point for most stores |
| **Thermal Printer** | Monospace, narrow-width design for 80mm/58mm thermal printers |
| **Gift Receipt** | Items without prices, customer note displayed as a gift message |

## Template Variables

Every receipt template receives two variables automatically:

### `$order` — The WooCommerce Order

This is the standard `WC_Abstract_Order` object. You can call any method you'd normally use in WooCommerce:

```php
$order->get_order_number();
$order->get_total();
$order->get_items();
$order->get_billing_email();
// … any WC_Order method
```

### `$receipt_data` — The Structured Receipt Payload

A pre-calculated array containing everything you'd typically need on a receipt, organised into logical sections. This is the recommended way to build new templates because the data is already formatted and consistent.

```php
$receipt_data['store']['name']                // Store name
$receipt_data['meta']['order_number']         // Order number
$receipt_data['lines']                        // Array of line items
$receipt_data['totals']['grand_total_incl']   // Grand total inc. tax
$receipt_data['payments'][0]['method_title'] ?? '' // Payment method (if present)
```

See the [Receipt Data Reference](./receipt-data-reference) for the complete schema with every field documented.

## Creating a Template

### From a Starter

1. Navigate to **WP Admin > POS > Templates**
2. In the **Starter Templates** section, click **Install**
3. The template opens in the editor — make your changes
4. Click **Activate** in the Template Actions panel to make it live

### From a Default Template

1. In the **Default Templates** section, click **Copy** next to the template you want
2. Edit the copy as needed
3. Activate when ready

### From Scratch

1. Click **Add New Template** at the top of the Templates page
2. Give it a name
3. Set **Template Type** to Receipt
4. Write your template code in the editor

## Template Engines

### PHP Templates (default)

Full PHP with access to all WordPress and WooCommerce functions. This is what the default template and all starters use.

```php
<?php
if ( ! defined( 'ABSPATH' ) ) {
    exit;
}

$store = $receipt_data['store'];
$lines = $receipt_data['lines'];
?>
<!DOCTYPE html>
<html>
<head>
    <style>
        body { font-family: sans-serif; font-size: 13px; }
    </style>
    <?php do_action( 'woocommerce_pos_receipt_head' ); ?>
</head>
<body>
    <h1><?php echo esc_html( $store['name'] ); ?></h1>

    <?php foreach ( $lines as $line ) : ?>
        <div>
            <?php echo esc_html( $line['name'] ); ?>
            — <?php echo esc_html( $line['qty'] ); ?>
            × <?php echo wp_kses_post( wc_price( $line['unit_price_incl'] ) ); ?>
        </div>
    <?php endforeach; ?>
</body>
</html>
```

:::warning Important
Always include `do_action( 'woocommerce_pos_receipt_head' )` in the `<head>` section. This injects the JavaScript that lets the POS app trigger printing.
:::

### Logicless Templates

A simpler option that uses `{{key.path}}` placeholders substituted directly from the `$receipt_data` array. No PHP, no programming — just HTML, CSS, and placeholder tags.

```html
<!DOCTYPE html>
<html>
<head>
    <style>
        body { font-family: sans-serif; font-size: 13px; max-width: 300px; margin: 0 auto; }
        .label { color: #666; }
    </style>
</head>
<body>
    <h1>{{store.name}}</h1>
    <p>{{store.address_lines}}</p>
    <p>{{store.phone}}</p>

    <hr>
    <p>Order #{{meta.order_number}}</p>
    <p>Date: {{meta.created_at_gmt}}</p>
    <p>Served by: {{cashier.name}}</p>
    <p>Customer: {{customer.name}}</p>

    <hr>
    <p><span class="label">Subtotal:</span> {{totals.subtotal_incl}}</p>
    <p><span class="label">Tax:</span> {{totals.tax_total}}</p>
    <p><strong>Total: {{totals.grand_total_incl}}</strong></p>
    <p><span class="label">Paid:</span> {{totals.paid_total}}</p>
    <p><span class="label">Change:</span> {{totals.change_total}}</p>

    <hr>
    <p>Thank you for your purchase!</p>
</body>
</html>
```

To use a logicless template, set the **Engine** to `logicless` in the Template Settings panel.

#### How It Works

The renderer flattens the `$receipt_data` array into dot-path keys and replaces every `{{key.path}}` placeholder with the matching value. Missing keys produce an empty string — no errors, no broken output.

- **Dot notation** maps directly to nested array keys: `{{store.name}}` → `$receipt_data['store']['name']`
- **Simple arrays** (like `store.address_lines`) are joined with commas automatically
- **Output is sanitised** through `wp_kses_post()` — safe by default

Every scalar value in `$receipt_data` is accessible via its dot path. Here's the full list:

:::info What's Not Available
Logicless templates cannot iterate over arrays. This means **individual line items**, **tax breakdowns**, **fee lines**, and **payment methods** are not directly accessible. If your receipt needs to list each product the customer bought, use a PHP template.

This is by design — logicless templates prioritise simplicity and offline safety over completeness.
:::

#### Limitations

Because there's no loop or conditional syntax, logicless templates are best suited for:

- **Summary receipts** — store info, order number, totals, and a thank-you message
- **Gift receipts** — store info and order reference without line-item detail
- **Quick receipts** — a minimal proof-of-purchase for the customer
- **Fiscal receipt headers/footers** — store details and fiscal compliance fields

They're not the right tool for detailed, itemised receipts. That's what PHP templates are for.

## Choosing an Engine

The two engines serve different needs. Pick the one that fits your situation.

| | PHP Templates | Logicless Templates |
|---|---|---|
| **Skill required** | Basic PHP and HTML | HTML and CSS only |
| **Line items** | Yes — full loop support | No — summary totals only |
| **Conditionals** | Yes — `if`/`else`, ternary | No |
| **Currency formatting** | Yes — `wc_price()` | No — raw numbers only |
| **Custom logic** | Anything PHP can do | Simple placeholder substitution |
| **Offline rendering** | No — requires server | **Yes — renders on the device** |
| **Security surface** | PHP execution (sandboxed) | No code execution at all |
| **Editing difficulty** | Moderate | Very low |
| **Best for** | Full itemised receipts | Summary receipts, offline environments |

### Why Offline Matters

POS systems live in the real world. Wi-Fi drops, servers go down, and the lunch rush doesn't wait. With PHP templates, the receipt has to round-trip to your WordPress server to render — if that connection is unavailable, no receipt.

Logicless templates flip that around. Because they're just HTML with placeholders and the receipt data is already on the device, the POS app can substitute the values and render the receipt locally. No network required.

For stores that operate in areas with unreliable internet, use mobile POS setups, or simply want a safety net for connectivity issues, logicless templates are worth the trade-off of not having itemised line items.

## Working with Currency

In PHP templates, use `wc_price()` to format currency values from `$receipt_data`:

```php
<?php echo wp_kses_post( wc_price( $receipt_data['totals']['grand_total_incl'] ) ); ?>
```

If you need plain text (no HTML tags), strip them:

```php
<?php
$currency = $receipt_data['meta']['currency'];
$amount   = $receipt_data['totals']['grand_total_incl'];
echo esc_html(
    html_entity_decode(
        wp_strip_all_tags( wc_price( $amount, array( 'currency' => $currency ) ) )
    )
);
?>
```

## Template Editor Features

The built-in editor provides:

- **Syntax highlighting** — PHP and HTML colour-coded with CodeMirror
- **Live preview** — See how your template renders with the most recent POS order
- **Revision history** — WordPress saves revisions each time you update, so you can roll back
- **Activate/Deactivate** — Switch between templates without deleting anything

## Default Template Sources

The system automatically detects filesystem templates from three sources, in priority order:

| Source | Location | Priority |
|--------|----------|----------|
| **Theme** | `your-theme/woocommerce-pos/receipt.php` | Highest |
| **Pro Plugin** | Ships with WCPOS Pro (requires active licence) | Medium |
| **Core Plugin** | Ships with WCPOS | Lowest |

If multiple sources exist, the highest-priority template is used as the default. You can always override this by activating a different template.

## Hooks and Filters

| Hook | Type | Description |
|------|------|-------------|
| `woocommerce_pos_receipt_head` | Action | Fires in `<head>` — injects print JavaScript |
| `woocommerce_pos_before_template_render` | Action | Fires before the template is rendered |
| `woocommerce_pos_after_template_render` | Action | Fires after the template is rendered |
| `woocommerce_pos_active_receipt_template` | Filter | Override which template is used |
| `woocommerce_pos_print_receipt_path` | Filter | Override the filesystem template path |

## Best Practices

1. **Start from a starter or copy** — modifying a working template is faster and safer than building from nothing
2. **Test with Preview** — use the preview button before activating; it renders with your most recent POS order
3. **Escape output** — always use `esc_html()`, `esc_attr()`, or `wp_kses_post()` when outputting data
4. **Keep CSS minimal** — receipt printers have limited rendering capabilities; avoid gradients, web fonts, and complex layouts
5. **Include the print hook** — without `do_action( 'woocommerce_pos_receipt_head' )`, the POS app can't trigger printing

### Getting Help

If you're stuck:

1. Check the template preview for error messages
2. Review the WordPress debug log (`wp-content/debug.log`) for PHP errors
3. Start with a starter template and modify incrementally
4. **AI tools** like ChatGPT, Claude, or Gemini are good at helping with straightforward WordPress PHP templates — paste your code and describe what you want
